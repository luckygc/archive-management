## Context

当前项目已经有档案元数据、动态表、item/volume、搜索投影、文件存储和登录认证等基础能力。能力地图中的 MVP 还要求真实档案可持续维护，包括全宗治理、档号唯一性、批量导入导出、审计、后端数据范围和档案电子文件管理。这些能力跨多个模块，不适合直接塞进一个代码变更。

## Goals / Non-Goals

**Goals:**

- 用 OpenSpec 固定 MVP 完成门槛。
- 把 MVP 缺口拆成可独立实施的后续 change。
- 明确哪些现有能力可计入 MVP 基础，哪些能力仍是缺口。
- 保持 MVP 简洁：优先支持真实维护、查询、导入导出、文件下载和审计。

**Non-Goals:**

- 不在本 change 一次性实现所有 MVP 代码。
- 不实现分类方案版本、完整档号规则引擎、正式归档流程、移交接收、四性检测、利用审批和长期保存。
- 不新增项目级抽象层、通用工作流引擎或独立 BFF。

## Decisions

### MVP 合同独立成 `archive-mvp-foundation`

MVP 是阶段完成门槛，不是单一业务模块。独立规格可以把跨模块验收口径集中表达，同时避免污染 `archive-metadata`、`file-storage` 等已有规格的职责。

替代方案是直接扩展各个现有规格，但这样会让“阶段门槛”和“模块合同”混在一起，后续判断 MVP 是否完成时需要在多个规格中拼接含义。

### 先固定手工档号，不做档号规则引擎

MVP 要求档案可引用、可校验，但不要求自动生成。首版使用 `archive_no` 固定字段和分类唯一规则表达唯一性；规则引擎、流水、补号和档号历史后续作为基础能力单独设计。

### 导入导出以查询结果和错误回填为边界

MVP 支持真实批量维护，最小能力是 Excel 导入、逐行错误回填和按查询条件导出。标准移交包、接收回执和跨系统重传属于接收/移交能力，不进入本阶段。

### 数据范围必须在后端执行

MVP 不能依赖前端隐藏按钮。后续实现应让管理查询、全文发现、详情读取、创建、更新、删除、下载和导出共享后端全宗/分类范围判断。

## Risks / Trade-offs

- 范围仍然偏大 → 使用后续独立 change 拆分代码实现，避免一个 change 同时改权限、导入导出和档案电子文件管理。
- 手工档号无法满足长期自动编号 → MVP 只保证可保存和可唯一，规则引擎推迟到基础能力。
- 导入导出格式容易过早标准化 → 首版只固定项目内标准 Excel 合同，不冒充电子档案移交包。
- 阶段规格可能和模块规格重复 → `archive-mvp-foundation` 只写完成门槛，具体字段和接口后续仍落到模块规格。

## Migration Plan

1. 先合并本 change，固定 MVP 缺口和验收门槛。
2. 后续按任务清单拆分独立 OpenSpec change 和代码实现。
3. 每个后续 change 完成后回看 `archive-mvp-foundation`，只在对应场景全部落地后勾选任务。
