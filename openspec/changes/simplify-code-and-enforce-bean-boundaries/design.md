## Context

项目已经按业务模块组织后端，并通过 ArchUnit 固化了包依赖、Repository、分页和 Spring 组件构造器规则，但尚未约束 Spring Bean 的 public 自调用和无收益业务接口。前端共享 API 与类型则集中在两个接近千行的 `archive.ts` 中。元数据与档案记录模块还存在总门面、专项 Service 和大量公开重载混合的问题。

本变更不改变外部业务合同，而是建立可执行约束后分批收缩内部边界。详细设计以 `docs/superpowers/specs/2026-07-14-code-simplification-and-spring-bean-boundaries-design.md` 为准。

## Goals / Non-Goals

**Goals:**

- 用 ArchUnit 阻止同一 Spring Bean 的 public 方法互调。
- 删除无真实多实现价值的 Service/Manager 接口，同时保留明确框架与稳定端口例外。
- 删除总门面与专项 Service 的重复入口，收缩巨型 Service。
- 验证并统一固定实体用户审计字段来源。
- 按业务域拆分前端 API 和类型，删除确定重复。

**Non-Goals:**

- 不修改 HTTP API、数据库迁移、权限或错误合同。
- 不引入通用 CRUD、Facade、Adapter、代码生成器或额外前端状态层。
- 不把动态 SQL 从 MyBatis 迁移到 Jakarta Data Repository。
- 不以文件数或行数作为独立完成目标。

## Decisions

### 规则先于修复

先编写会失败的 ArchUnit 测试，使用测试输出形成真实违规清单，再逐项修复。相比先重构后补规则，这能证明规则确实覆盖现有问题，并在迁移期间阻止新增违规。

### public 入口共享 private 核心实现

同一 Bean 的多个 public 用例如果只是在默认参数或入口语义上不同，共同调用 private 核心方法。只有当复用逻辑具有独立事务、权限或业务边界时才拆成另一个具体 Spring Bean。拒绝 self 注入和代理查找，因为它们隐藏依赖并规避静态结构。

### 只自动检查业务 Service/Manager 接口

单实现接口规则限定在业务模块的 Service/Manager 范围，并显式排除 Repository、Mapper、稳定基础设施端口和已有多实现策略。相比扫描所有 Java 接口，这避免把框架声明式接口和领域标记接口误判为过度抽象。

### 分批移除门面

元数据和档案记录能力按完整业务用例迁移调用方；一批迁移完成后同步删除旧门面入口，不长期维护双入口。相比一次性拆完所有类，此方案缩小每次回归面；相比永久保留兼容门面，它能真正减少依赖与公开表面。

### 审计删除必须由集成测试先证明

手工审计字段不能仅凭存在拦截器就删除。先运行并强化无状态 Repository 审计测试，确认 insert/update 路径使用安全上下文；MyBatis、业务操作人和未经过拦截器的路径保留显式赋值。

### 前端只按业务域拆分

API 函数和类型移动到业务域文件，继续直接调用 `frontend-core` HTTP client。不增加通用 API 工厂或 Repository 层，因为当前重复主要来自文件聚合和局部工具重复，而不是缺少运行时抽象。

## Risks / Trade-offs

- [ArchUnit 对编译器生成或 lambda 调用的建模可能与源码直觉不同] → 先用已知违规验证失败输出，并为规则自身增加聚焦测试样本。
- [拆除门面导致大量导入和构造器变化] → 按业务闭环迁移 Controller、跨模块调用与测试，每批保持编译通过。
- [public 重载可能是外部调用合同] → 修改可见性前搜索生产和测试调用方，仅把确属本类内部复用的入口改为 private。
- [删除手工审计赋值可能影响无安全上下文测试或启动初始化] → 按写入路径分类，只有被集成测试证明由拦截器覆盖的路径才删除。
- [前端文件移动造成循环依赖] → 公共分页与 HTTP 类型保持在窄小公共文件中，业务域文件之间不互相导入页面代码。

## Migration Plan

1. 更新协作规则并增加架构测试，记录现有失败清单。
2. 修复 public 自调用和单实现业务接口违规。
3. 按业务闭环迁移并删除元数据、档案记录重复门面入口。
4. 验证后删除固定实体重复审计赋值。
5. 拆分前端 API/type 文件并更新调用方。
6. 运行后端完整测试及前端 check、test、build。

所有改动均为源码级内部迁移，不需要运行期双写或数据库回滚。回退时按独立提交恢复对应批次。
