## Why

现有“本体/字段语义 + 本地规则”模型仍保留抽象字段注册、任意字符串触发点和手工执行接口，但没有进入档案写入事务，因此不同机构基于自有字段配置的约束和业务规则不能形成不可绕过的运行时保障。需要收敛为直接引用真实固定字段与分类字段的运行时约束和规则，由用户定义条件、作用域与参数，由系统提供固定触发点和固定动作，并让整套动态运行时配置可以快速导出、迁移和恢复。

## Goals

- 从现行生产代码、API、菜单、权限文案、稳定规格和知识库现状说明中移除“档案本体”及其派生的字段语义注册模型。
- 提供用户可配置的结构化运行时约束和规则，并在真实档案写入、组卷、文件上传和导出事务边界触发。
- 系统提供版本化的固定触发点和固定动作目录；首批动作只包含阻断、警告和写入候选字段值，动作参数按字段类型校验。
- 用 PostgreSQL `NOT NULL`、`CHECK`、外键、唯一/排斥约束以及少量 Flyway 管理的触发器守住可由数据库判断的不变量。
- 提供独立于业务数据和数据库 ID 的运行时配置快照，用于跨环境迁移、快速备份和原子恢复。

## Non-Goals

- 不允许管理员提交 SQL、PL/pgSQL、SpEL、JavaScript 或其他可执行脚本，也不把动态检查编译为数据库函数或触发器。
- 不允许规则直接写数据库、启动流程、发布事件、调用远程服务或操作文件；`SET_FIELD` 只修改当前事务尚未持久化的候选字段值。
- 不新增尚无真实业务入口的归档、移交、销毁或长期保存状态机触发点。
- 不建设 PostgreSQL 整库备份、WAL/PITR、对象存储灾备或应用发布回滚能力；本 change 的备份恢复仅指用户定义的动态运行时配置。

## What Changes

- **BREAKING** 删除字段语义资源、旧本体残留类型和 `/api/v1/archive-field-semantics`，规则条件改为直接引用受控固定字段、当前分类动态字段、实物字段和上下文字段。
- **BREAKING** 将泛化本地规则重构为两种用户配置：运行时约束表达“候选数据必须满足的断言”，运行时规则表达“`WHEN` 结构化条件命中后执行系统固定动作”；移除任意 `triggerCode` 和自由 effect。
- 固定触发点由代码枚举，并接入档案条目创建/修改/删除、案卷创建/加入条目、电子文件上传和档案导出 Service；导入复用条目创建/修改链路，不维护第二套触发器。
- 固定动作由系统注册并声明参数合同与适用触发点；首批提供 `REJECT`、`WARN` 和 `SET_FIELD`，规则不得提交未注册动作。
- 条件与动作直接使用当前作用域可见的固定字段、分类动态字段、实物字段和上下文字段；发布时按真实字段定义校验存在性、数据类型、可写性和操作符。
- 管理端以“运行时规则”工作区替代“字段语义/本地规则”入口，提供约束/规则草稿编辑、字段选择、固定动作配置、发布前校验、启停、试运行和可定位失败反馈。
- 新增 PostgreSQL 数据不变量清单及迁移：普通约束优先；仅对发布后不可变和必须在事务结束检查的跨表状态使用版本化触发函数/约束触发器。
- 新增运行时配置快照导出、摘要校验、预检、原子导入和草稿全量恢复；快照只包含治理作用域、运行时约束、规则、固定动作参数及真实字段依赖。
- 当前版本为 `0.0.1`，按未发布目标结构直接重写治理基础 Flyway，删除本体表、旧规则/effect 表、相关历史 change 和兼容迁移，不保留旧模型数据或双读分支。

## Capabilities

### New Capabilities

- `archive-runtime-check-portability`: 定义运行时约束与规则配置包的版本、字段引用映射、完整性校验、预检、原子导入和跨环境迁移合同。

### Modified Capabilities

- `archive-ontology-core`: 删除字段语义注册及所有现行本体合同，不再保留替代性本体资源。
- `archive-local-rule-engine`: 将泛化规则/effect 和手工执行模型改为用户定义条件、系统固定触发点与固定动作的运行时约束和规则。
- `archive-governance-scheme`: 治理版本不再描述或解释本体，发布时校验运行时检查，历史档案仅追溯治理配置版本。
- `archive-governance-workbench`: 工作台改为维护运行时检查装配与解析，不再显示本体或字段语义入口。
- `archive-record-routing`: 条目和案卷状态变更在同一事务内执行对应运行时检查并处理阻断/警告结果。
- `archive-import-export`: 导入复用逐条写入检查，导出在生成文件前执行运行时检查。

## Impact

- 真相源：`openspec/specs/archive-ontology-core`、`archive-local-rule-engine`、`archive-governance-scheme`、`archive-governance-workbench`、`archive-record-routing`、`archive-import-export`、`api-contract`，以及 `docs/database.md`、`docs/ops-runbook.md` 和档案知识库现状说明。
- 后端：删除 `module/archive/ontology` 残留和字段语义资源，新增基于真实元数据的字段目录，重构约束/规则实体、Repository、Mapper、固定动作执行器、Service、Controller，并接入条目、案卷、电子文件和导出 Service。
- 数据库：直接维护 `0.0.1` 目标 Flyway，建立运行时定义、动作、追踪、约束和固定触发函数，删除旧本体与泛化规则结构。
- API：删除字段语义接口；运行时检查 CRUD、发布、试运行以及配置包导出/预检/导入遵循现有 `api-contract` 和 ProblemDetail。
- 前端：删除旧本体/字段语义页面、路由、类型和 client；重做运行时规则工作区、字段选择、固定动作参数表单、发布校验、试运行和配置包迁移反馈。
- 运行时配置：新增可下载快照、字段映射预检、原子导入/恢复和前端错误反馈，不涉及整库与文件内容备份。
