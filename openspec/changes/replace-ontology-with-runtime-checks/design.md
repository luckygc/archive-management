## Context

当前仓库已经开始删除五类本体资源，但仍保留字段语义注册、`ArchiveOntology*` 类型、任意字符串 `triggerCode`、泛化 effect 和手工 `:execute` 接口。规则执行器可以解析条件，却没有被档案条目、案卷、文件或导出 Service 调用；用户配置不会约束真实写入。

不同机构的分类和动态字段不同，因此运行时能力不能依赖一套预设行业本体。系统已有稳定的固定字段、按分类生成的动态字段和治理版本，可以直接形成带类型的字段目录。项目当前为 `0.0.1` 未发布阶段，数据库和规格可以按最终目标结构直接收敛，不需要兼容或迁移旧本体数据。

PostgreSQL 的 `CHECK` 适合同一行内不变量，外键、唯一约束和排斥约束适合数据库可直接判断的关系；跨行或需要事务结束再判断的不变量才使用约束触发器。用户定义的动态运行时配置通过应用层版本化快照迁移，不依赖整库工具。

## Goals / Non-Goals

**Goals:**

- 用户按机构、治理版本、全宗、分类和固定触发点定义运行时约束与规则。
- 条件直接引用真实固定字段、分类动态字段、实物字段和系统上下文，并在发布与执行时做类型校验。
- 系统只执行代码内注册的固定动作，第一批为 `REJECT`、`WARN`、`SET_FIELD`。
- 规则进入真实 Service 事务，在持久化前确定性执行，阻断结果回滚整个业务操作。
- 动态运行时配置可独立于业务数据快速导出、备份、跨环境迁移和恢复。

**Non-Goals:**

- 不提供 SQL、PL/pgSQL、SpEL、MVEL、JavaScript、Groovy 或其他脚本入口。
- 不让用户规则直接访问 Repository、Mapper、HTTP、Spring Bean、文件系统或远程服务。
- 不把每条用户规则生成为 PostgreSQL `CHECK`、函数或触发器。
- 不预设尚未存在的移交、销毁、归档审批等业务触发点。
- 不建设 PostgreSQL 整库备份、WAL/PITR、对象存储恢复或旧本体数据兼容迁移。

## Decisions

### 决策 1：以真实字段目录替代本体和字段语义注册

运行时字段目录由系统在请求时按触发点和分类组合：

- 固定字段使用稳定代码，例如 `item.archiveNo`、`item.archiveYear`、`item.securityLevelId`。
- 分类元数据字段使用 `metadata.<fieldCode>`，实物字段使用 `physical.<fieldCode>`。
- 只读上下文字段使用 `context.userId`、`context.now`、`context.operation`。

每个目录项包含字段代码、名称、数据类型、可读/可写标记、适用触发点和来源分类。发布时解析所有引用并保存定义快照摘要；执行时重新按当前治理版本和档案分类解析，字段已删除、停用或类型不兼容时按配置失效处理并阻断业务操作，不能悄悄跳过。

跨环境配置包使用分类编码和字段编码，不使用数据库 ID。导入目标环境必须完成一一映射和类型兼容校验。

替代方案是继续维护字段语义表。它会让用户同时维护真实字段和第二套映射，并再次产生“映射已配置但业务未执行”的分叉，因此不采用。

### 决策 2：明确区分运行时约束与运行时规则

统一持久化根 `am_archive_runtime_definition` 保存公共字段：治理版本、`CONSTRAINT`/`RULE` 类型、稳定编码、名称、触发点、作用域、优先级、结构化条件、状态、启用标记、发布信息和审计字段。

- 约束的条件表达必须成立的断言；失败处理只允许 `REJECT` 或 `WARN`，并保存可展示消息。
- 规则的条件表达 `WHEN`；命中后按顺序执行 `am_archive_runtime_action` 中的固定动作。
- `am_archive_runtime_action` 保存动作类型、顺序和结构化参数，不保存脚本或类名。
- 条件 AST 只允许 `all`、`any`、`not`、字段比较和空值判断；深度、节点数、集合参数数和字符串长度均有上限。

定义在同一治理版本内按编码永久唯一。发布后的条件、作用域和动作不可原地修改；启停是唯一允许的运行态开关，并记录操作者与时间。

替代方案是把约束也表达为带 `REJECT` effect 的普通规则。它无法清楚表达“断言必须为真”，管理界面和追踪也难以区分数据约束与条件动作，因此不采用。

### 决策 3：触发点和动作都由系统固定注册

第一批触发点只覆盖仓库已有真实 Service：

- `ITEM_BEFORE_CREATE`
- `ITEM_BEFORE_UPDATE`
- `ITEM_BEFORE_DELETE`
- `VOLUME_BEFORE_CREATE`
- `VOLUME_BEFORE_ADD_ITEM`
- `FILE_BEFORE_UPLOAD`
- `EXPORT_BEFORE_CREATE`

导入逐行调用条目创建或修改，不新增 `IMPORT` 触发点。固定动作目录为：

- `REJECT`：形成阻断决策，所有规则完成后统一返回可定位错误并回滚。
- `WARN`：形成非阻断决策并写入规则追踪；有响应体的写操作通过对应响应视图返回警告。
- `SET_FIELD`：只允许在创建/修改类前置触发点修改当前内存中的候选固定字段、动态字段或实物字段；目标必须可写且值可转换为目标类型。

每个动作处理器由代码声明动作类型、参数合同、允许的触发点和可写字段范围。规则只保存动作类型和参数，不能指定 Bean、类名或函数。删除、文件上传和导出触发点不允许 `SET_FIELD`。

替代方案是允许用户注册动作脚本或数据库函数。它扩大为远程代码执行和数据库权限边界，无法安全迁移、审计和回滚，因此不采用。

### 决策 4：采用单次、确定性的事务内执行管线

每个接入 Service 在数据库状态变更前调用具体的 `ArchiveRuntimePolicyService`：

1. 根据档案记录或候选请求解析治理版本、触发点、全宗、分类和字段目录。
2. 将当前值与请求值合并为完整候选事实，完成基础类型转换。
3. 查询 `PUBLISHED` 且启用的定义，按 `priority ASC, definition_code ASC, id ASC` 排序。
4. 顺序执行规则。`SET_FIELD` 只修改候选事实，不直接持久化；两个命中规则给同一字段设置不同值时产生配置冲突并阻断，同值重复设置视为幂等。
5. 在规则动作完成后统一执行约束，收集全部 `REJECT` 和 `WARN` 决策。
6. 有阻断或配置失效时抛出项目 ProblemDetail，当前事务不写主数据、动态表、文件元数据、审计或追踪；无阻断时由原 Service 使用最终候选值完成持久化，并在同一事务写入必要追踪。

规则不因 `SET_FIELD` 重新从头迭代，因此不存在循环执行；后续规则可以读取先前规则已写入的候选值。执行数量、AST 大小和单次耗时设置硬上限，超限按配置故障关闭写入，不降级为跳过。

替代方案是通过数据库行触发器调用动态规则。数据库触发器看不到完整权限、动态字段候选值和文件上下文，也难以返回一致 ProblemDetail，因此只用于数据库自身不变量。

### 决策 5：PostgreSQL 原生约束优先，触发器保持少而固定

Flyway 为运行时定义表建立命名约束：

- `NOT NULL`、长度 `CHECK`、状态/类型/触发点/动作白名单 `CHECK`。
- 治理版本和定义/动作的外键。
- 活动定义编码唯一索引及稳定查询索引。
- `jsonb_typeof(...) = 'object'` 的形态约束；深层 AST 和动作参数由应用发布校验负责。

只新增两个版本化的 PL/pgSQL 边界：

- `BEFORE UPDATE OR DELETE` 不可变触发器：父定义发布后拒绝修改定义字段或删除，启停与审计列除外；发布规则的动作行同样不可改删。
- `DEFERRABLE INITIALLY DEFERRED` 约束触发器：事务提交前保证发布规则至少有一个动作、约束没有动作，且非草稿治理版本只包含已发布定义。这样覆盖同一事务内先写子表再发布父表的合法顺序。

触发函数和触发器只由 Flyway 创建，使用固定名称和异常 SQLSTATE/约束名；不硬编码 `public` schema。Service 将已知约束名映射为稳定 ProblemDetail，未知数据库错误保持系统错误并携带 `traceId`。

### 决策 6：发布前校验和试运行使用同一执行核心

发布动作校验字段引用、操作符、条件参数、动作参数、触发点兼容性、作用域、定义数量和治理版本可编辑状态。试运行接收显式候选事实，但服务端仍从字段目录校验字段和类型，返回执行顺序、条件命中、候选字段变更、警告和阻断；试运行不写主数据或追踪。

管理 API 继续遵循 `/api/v1`、完整 Controller URL、AIP-136 冒号动作和 ProblemDetail。运行时定义使用标准资源 CRUD；发布、启用、停用、试运行及配置包预检/导入使用专用动作请求/响应，不使用统一 `Result<T>`。

### 决策 7：单一运行时配置快照负责迁移、备份和恢复

运行时配置快照是规范化 JSON 文档，包含 `schemaVersion`、来源应用版本、导出时间、治理方案/版本稳定编码、作用域、定义、动作、字段引用和 SHA-256 摘要。它不包含档案记录、文件、用户、权限、审计或数据库 ID。导入与恢复共用同一预检：

1. 校验文档版本、摘要、大小和节点上限。
2. 用目标环境的分类编码和字段编码解析引用，报告缺失、重复和类型不兼容。
3. 普通跨环境导入在单个数据库事务中创建新的 `DRAFT` 治理版本和草稿定义；任何失败全部回滚。
4. 快速恢复对管理员明确选择的 `DRAFT` 治理版本执行全量替换；先删除该草稿现有运行时定义，再写入快照内容，整个过程处于同一事务，失败时恢复前草稿保持不变。
5. 导入或恢复均不自动发布、不修改已发布版本；管理员复核并试运行后再发布。

配置快照既是迁移载体，也是动态运行时部分的备份文件。系统提供导出、预检、导入和恢复 API/页面；不调用 `pg_dump`、`pg_restore`，不处理对象存储或业务数据。快照下载文件名包含治理稳定编码和导出时间，内容摘要用于发现损坏或篡改，但不替代权限校验。

## Failure Paths

- 规则引用字段缺失、停用或类型变化：发布/导入预检失败；若已发布配置在运行时失效，则关闭对应写操作并返回配置错误。
- 多条规则冲突写同一字段：记录冲突规则和字段，阻断事务；不以“最后一条覆盖”隐藏错误。
- 规则超限或执行器异常：事务回滚，不跳过规则继续写入；日志和 ProblemDetail 保留定义编码及 `traceId`，不回显敏感候选值。
- PostgreSQL 约束或触发器拒绝写入：已知约束映射业务错误，未知错误保持系统错误；事务不得部分提交。
- 配置包不兼容：预检返回逐项错误，不创建草稿；原配置保持不变。
- 配置快照恢复失败：整个事务回滚，目标草稿恢复前内容保持不变；修正字段映射或选择其他快照后重试。

## Risks / Trade-offs

- [运行时规则增加写路径延迟] → 按治理版本、触发点和作用域批量读取并缓存已发布定义；发布/启停后精确失效，执行数和耗时有指标与硬上限。
- [动态字段变化使已发布规则失效] → 发布保存字段签名，字段删除或改型前反向检查引用；运行时再次校验并失败关闭。
- [用户期望任意业务自动化] → 固定动作目录只列已接入且可事务化的动作；新增动作必须独立扩展合同、权限、失败和恢复测试。
- [警告响应会影响现有 HTTP 视图] → 仅在确需展示警告的命令响应增加专用决策字段，不引入全局响应包装。
- [数据库触发器降低排障透明度] → 仅保留两个固定职责，统一命名、SQLSTATE、迁移来源和 PostgreSQL 集成测试，不把用户定义放进触发函数。
- [快照字段依赖与目标环境不一致] → 快照保存稳定编码和类型签名，恢复前必须完整预检，不允许部分映射或静默跳过。
- [恢复误覆盖草稿工作] → 只允许显式选择 `DRAFT` 目标并展示替换摘要；后端以单事务全量替换，已发布版本永不覆盖。

## Migration Plan

1. 直接重写尚未发布的治理 foundation Flyway 为运行时定义/动作/追踪目标结构，删除本体、字段语义和旧规则/effect DDL，不编写旧数据转换逻辑。
2. 删除旧本体 Java/Vue/API、测试、知识库历史说明、旧迁移和相关已归档 OpenSpec change，不保留兼容路由或双读。
3. 后端实现字段目录、发布校验、固定动作执行器和试运行 API；暂不接业务写入。
4. 接入真实 Service 并以 PostgreSQL 事务集成测试证明阻断回滚、字段赋值、警告追踪和冲突失败。
5. 前端切换到运行时规则工作区，删除本体、字段语义和旧规则入口/API client。
6. 增加运行时配置快照导出、预检、导入和草稿全量恢复，在两个不同字段 ID 的环境间完成迁移与恢复测试。

由于当前为 `0.0.1`，开发数据库直接重建并由 Flyway 生成目标结构；不提供从旧本体结构升级或回滚的生产路径。

## Open Questions

- 无阻塞问题。首批固定动作限定为 `REJECT`、`WARN`、`SET_FIELD`；任何流程、通知、编号生成或外部调用动作均按后续真实业务 change 扩展。
