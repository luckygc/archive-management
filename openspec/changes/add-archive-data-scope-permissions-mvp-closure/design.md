## Context

项目当前已有登录认证、角色、档案元数据、动态表、档案记录查询、全文发现、文件存储、档案电子文件管理和基础审计。MVP 仍缺三类闭环能力：后端数据范围、可枚举/可判断的功能权限、标准导入导出。现有授权模块只有角色和用户角色关系，Spring Security 只负责登录态和少量基础角色判断；档案记录查询主要由 MyBatis 拼接动态表 SQL。

数据范围与功能权限需要拆开：功能权限回答“用户能做什么动作”，数据范围回答“哪些档案属于用户范围”。运行时统一组合为“已登录 + 有功能权限 + 目标数据命中数据范围”。数据范围本身不包含 READ/WRITE/DELETE 语义。

## Goals / Non-Goals

**Goals:**

- 支持独立档案数据范围资源，包含 `*` 任意范围、全宗、分类、密级和分类内动态字段条件。
- 动态字段条件只在数据范围定义的 jsonb 部分保存；不保存 SQL，不为权限表做动态 DDL。
- 支持权限点枚举、角色权限绑定和当前用户权限判断接口。
- 档案管理查询、全文发现、详情、写入、档案电子文件上传、列表、预览、下载、删除、审计查询、导入导出统一应用数据范围；档案电子文件列表可见、预览和下载使用不同功能权限。
- 补齐标准 Excel 导入导出，使 MVP 能支撑真实批量维护。

**Non-Goals:**

- 不做字段级脱敏、利用审批、密级流转、部门组织树继承、流程节点权限或数据库 RLS。
- 不做任意布尔表达式规则引擎，不允许管理员输入 SQL。
- 不把动态字段复制到权限表，也不因动态字段变更迁移权限表结构。
- 不把数据范围绑定到 READ/WRITE 动作；读写差异通过角色的功能权限组合表达。

## Decisions

### 数据范围独立于功能权限

新增 `archive-data-scope` 能力。数据范围表只描述档案集合，范围通过通用主体关系表绑定到授权主体。MVP 支持 `ROLE` 和 `USER`，并预留 `ORG_UNIT` 主体类型给后续组织架构接入。角色功能权限由独立权限点关系维护。运行时取当前用户直接绑定范围、启用角色范围的并集；后续组织范围在组织模块提供用户所属组织后接入同一主体关系表。

备选方案是在数据范围中加入 READ/WRITE，但这会让范围同时承担动作语义，导致“读全量、写部分”这类需求难以复用。另一个备选是只通过角色绑定范围，但部门范围会导致角色膨胀，因此不采用。

### `*` 作为范围类型

任意范围使用 `scope_type = ALL` 表达，语义等价于 `*`。`ALL` 范围不能配置任何条件，命中后数据范围谓词为 `true`。无绑定范围默认拒绝，不隐式等价于 `*`。

备选方案是使用特殊全宗或分类 ID 表示全量，但会污染业务维度，也容易和真实数据冲突。

### 固定维度结构化，动态字段条件用 jsonb

数据范围固定维度使用结构化列或结构化子表保存，包括全宗、分类、密级和是否包含子级。动态字段条件保存在 `dynamic_condition_json jsonb` 中，形如分类 ID、字段编码、操作符和值集合。保存时后端校验分类、字段、类型、操作符和值；查询时后端解析 JSON 并编译为受控 MyBatis 条件。

备选方案是全部条件都放 jsonb，开发最快但失去固定维度的数据库约束和查询便利。另一个备选是权限表动态 DDL 加字段，该方案会把档案动态元数据污染到授权配置表，迁移成本高，因此不采用。

### 范围一致性主要由服务层校验

数据库只保留必要的主键、外键、唯一索引、非空和基础检查，不使用触发器表达 `ALL` 与条件互斥、条件范围不能为空、动态字段 JSON 内容合法等跨表或跨字段业务规则。这些规则由 `ArchiveDataScopeService` 在创建、更新和启停时统一校验，并由测试覆盖。

备选方案是使用触发器或复杂约束在数据库层兜底，但这会增加迁移、排障和回滚成本；当前项目尚未正式发布，且数据范围配置入口受控，服务层校验更符合 MVP 的简洁边界。

### 查询执行使用范围编译器

新增 `ArchiveDataScopeService` 计算当前用户有效范围，输出内部 `ArchiveDataScopePredicate`。MyBatis 只接收后端构造的谓词对象，不接收前端 SQL 片段。查询条件顺序为：

```text
deleted_flag = false
AND 用户数据范围谓词
AND 用户查询条件
```

写入、下载和导出通过同一服务判断目标档案是否落在范围内，不能只在列表查询时过滤。

### 功能权限使用稳定权限点

新增稳定权限点，例如 `archive:item:read`、`archive:item:create`、`archive:item:update`、`archive:item:delete`、`archive:item:lock`、`archive:item:preview-electronic-file`、`archive:item:download-electronic-file`、`archive:export`、`archive:metadata:manage`、`archive:data-scope:manage`、`authorization:permission:manage`。权限点可枚举，角色可绑定，当前用户可查询自身权限并由后端判断接口使用。导入不设置独立功能权限，按每行实际写入语义分别使用创建或编辑权限。

Spring Security 继续负责认证和少量基础入口保护；业务权限判断在业务 Service 层集中执行，以便结合档案数据范围。

### 导入导出进入同一授权链路

导入先按每行是否命中已有档案判断创建或编辑权限，并校验目标分类、全宗、动态字段和数据范围。存在错误时整个批次不写入，返回逐行错误。导出使用当前查询条件、当前用户数据范围和导出权限生成结果，并写入操作审计。

## Risks / Trade-offs

- 动态字段条件存在 jsonb 中缺少外键保护 → 保存和更新范围时强校验；字段删除或禁用时阻止删除被引用字段，或将范围标记为失效。
- 范围条件过多导致 SQL 变大 → 限制单个范围的固定维度和动态字段条件数量，查询时对同字段 `EQ/IN` 做交集收窄。
- 多主体多范围 OR 条件影响性能 → 优先命中 `ALL` 快速返回；固定维度建立索引；动态字段仅允许可搜索字段参与范围。
- 功能权限和数据范围组合容易漏接接口 → 在档案 Service 层增加统一检查，并用越权读写测试覆盖列表、详情、写入、下载、导出。
- Excel 导入导出格式过早泛化 → MVP 只支持项目内标准模板，不处理正式移交包或跨系统交换包。

## Migration Plan

1. 新增授权权限点、角色权限关系、档案数据范围、范围固定维度项、数据范围主体关系等表。
2. 初始化系统管理员角色的全部权限和 `*` 数据范围。
3. 为动态字段新增“允许用于数据范围”标记，默认关闭，由管理员显式启用。
4. 接入查询、详情、写入、下载和审计查询的数据范围判断。
5. 增加导入导出 API、服务和前端入口。
6. 回滚时可先撤下前端入口和角色绑定，保留迁移表不影响旧主链路。

## Open Questions

- 密级和保管期限按固定字典落库，档案主表只保存 `security_level_id`、`retention_period_id`。
- 导入导出是否同步完成即可满足 MVP；若大批量性能不足，再独立引入异步 job，不在本轮默认引入。
