## Context

当前字段定义已经承担动态表列、列表显示和搜索配置。控件属于字段自身配置，可以随字段定义维护；表格、详情和编辑布局属于不同使用场景，不能继续混在字段弹窗里配置。详情页需要可多开，编辑页需要独立保存状态，因此记录详情和编辑不应继续放在列表抽屉中。

## Goals / Non-Goals

**Goals:**

- 记录详情页通过路由打开，展示固定字段、动态字段、锁定状态和基础操作入口。
- 记录编辑页通过路由打开，编辑固定字段、全宗和动态字段。
- 字段定义支持编辑控件配置：`INPUT`、`TEXTAREA`、`NUMBER`、`DATE`、`DATETIME`。
- 表格、详情和编辑布局支持所见即所得拖拽配置，并分别维护显示、列宽、跨列和布局位置。
- 布局配置只支持公共布局，不提供个人布局。
- 字段定义维护基础列表列宽，作为表格列宽默认值。
- 长文本字段可以配置为 `TEXTAREA`，前端以多行文本控件编辑。
- 更新记录时校验业务锁；已锁记录不能编辑。
- 更新记录后写入全文检索投影 outbox 并写入档案记录操作审计。

**Non-Goals:**

- 不实现枚举选项、远程选择、人员、组织、文件上传等高级控件。
- 不支持编辑时切换档案分类。
- 不在本变更中实现附件管理、案卷管理和流程审批动作。

## Decisions

### 字段控件作为字段定义合同

`am_archive_field.edit_control` 保存编辑控件。默认控件按字段类型推导：

- `TEXT` 默认 `INPUT`，长文本业务字段可配置 `TEXTAREA`；
- `INTEGER`、`DECIMAL` 默认 `NUMBER`；
- `DATE` 默认 `DATE`；
- `DATETIME` 默认 `DATETIME`。

后端校验控件与字段类型兼容，避免日期字段配置为多行文本、数字字段配置为日期控件等无效组合。

### 布局配置独立于字段弹窗

表格、详情和编辑分别维护布局项，避免一个字段排序同时影响三种工作区。`am_archive_field_layout` 只保存公共布局，不保存 `owner_user_id`，也不提供“我的布局”保存入口。读取有效布局时直接读取公共布局；公共布局不存在时回退到字段定义中的默认展示配置。

字段弹窗只维护字段编码、名称、类型、控件、检索和状态等字段属性；分类字段维护页提供独立布局配置区，用户切换表格、详情、编辑后在预览画布中拖拽字段并维护该布局的属性。前端按有效布局渲染档案列表、详情和编辑页，不从字段定义排序推导页面布局。

### 字段基础列宽和布局覆盖边界

`am_archive_field.list_width` 是字段本身的基础表格列宽，作为档案列表渲染的默认列宽，可在字段定义中维护。布局表主要保存公共布局的可见性、顺序、跨列和布局位置；只有确有按布局覆盖列宽的业务需要时，才在布局项中保存表格列宽覆盖值。没有覆盖值时使用字段基础列宽。

### 布局配置交互与拖拽库

布局配置界面采用左侧字段池、右侧布局工作区。右侧按 `table`、`detail`、`edit` 分 Tab：

- 表格布局支持字段拖拽排序、显示开关和列宽编辑。
- 详情布局支持拖拽排序、跨列和显隐。
- 编辑布局支持拖拽排序、跨列和显隐。

拖拽库首选重新评估 Atlassian `pragmatic-drag-and-drop`，因为它框架无关、适合字段列表和表单/表格布局配置。`gridstack` 只适合自由 dashboard/grid 面板，不作为字段列表和表单/表格列配置默认选择。`@dnd-kit` 主要面向 React 生态，不作为本 Vue 项目默认选择。Vue 生态可比较 `vue-draggable-plus` / SortableJS 的维护状态和 Element Plus 配合情况。

### 详情和编辑使用独立路由

档案记录详情页使用 `/archive/library/:id`，编辑页使用 `/archive/library/:id/edit`。列表页只承担查询、选择和进入详情/编辑，不承载完整编辑状态。多页签场景下，详情和编辑天然以完整路径区分实例。

### 更新接口维护派生数据

`PATCH /api/v1/archive-records/{id}` 更新固定字段、动态表字段和全宗。更新前校验记录未锁定；更新后写入全文检索投影 outbox，并以 `UPDATE` 操作写入 `am_archive_record_audit`。动态字段转换复用创建记录的字段类型校验。

## Risks / Trade-offs

- 当前画布只表达字段项顺序和跨列，不表达复杂分组和嵌套区域 → 首版先满足表格/详情/编辑各自所见即所得排序和显示，复杂分组后续再引入区域模型。
- 编辑页和新增抽屉共用控件逻辑容易分叉 → 前端应抽出动态字段控件渲染组件。
- 控件配置变更可能影响已有记录编辑体验 → 控件只影响前端渲染，不改变动态表物理类型。
