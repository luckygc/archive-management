## Why

现有审批流定义只能在弹窗表格中维护顺序节点，无法以拖拽、连线和节点属性的方式表达或校验流程；同时审批模块又复制了 Flowable 的任务、候选人和意见数据，仍不能承载借阅、移交、质检等跨业务统一待办。现在需要把设计期、运行期和跨业务查询投影的责任拆清，并一次形成可发布、可办理、可审计的生产闭环。

## What Changes

- 新增独立的可视化审批流程设计页面，使用 Apache-2.0 的 LogicFlow 画布，支持开始、审批、条件分支和结束节点的拖拽、连线、选择、删除、撤销重做、缩放、适应画布、属性编辑和发布前校验。
- 将审批定义从顺序节点列表升级为带布局坐标、节点业务属性和条件连线的流程图；保存草稿时保留画布布局，发布时冻结版本并转换为 Flowable BPMN。
- 新增跨业务统一待办能力，以一名办理人一条投递记录的形式汇总审批及后续其他业务待办，提供我的待办、已办和来源跳转信息。
- **BREAKING**：审批任务列表改为统一待办投影；审批任务、候选关系和审批意见不再以项目自有表作为流程真相源，分别复用 Flowable 任务、IdentityLink、历史和 Comment API。
- 精简审批持久化，只保留定义草稿、发布版本、业务实例绑定和统一待办四类项目表；统一待办是可重建的查询投影，办理权限仍由来源业务和 Flowable 校验。
- 保持业务模块只能调用审批和统一待办公开服务，不允许直接依赖 Flowable API 或原生表。

### 目标

- 管理员可以在 PC 工作台完成流程绘制、配置、校验、保存、发布和版本冻结。
- 经办人在一个统一入口查看不同业务来源的待办与已办，并回到正确的来源业务办理。
- Flowable 成为审批运行时、候选关系和审批意见的唯一流程真相源，消除双写同步负担。

### 非目标

- 不接入 Flowable Modeler、IDM、CMMN、DMN 或 Flowable 商业套件。
- 不支持任意脚本、SpEL、UEL 或用户直接编辑 BPMN XML；条件仅使用受控字段、运算符和值模型。
- 不实现并行网关、会签、加签、转办、委托、超时和运行中版本迁移。
- 不主动启动长期占用端口的开发预览服务。

## Capabilities

### New Capabilities

- `unified-todo`: 跨业务待办投递、我的待办/已办查询、来源跳转、来源完成关闭和服务端二次授权。

### Modified Capabilities

- `approval-workflow`: 将顺序节点表单升级为可视化流程图和受控条件分支，发布为 Flowable BPMN，并改用统一待办投影及 Flowable 原生任务、候选关系、历史和意见能力。

## Impact

- 真相源：修改 `openspec/specs/approval-workflow/spec.md`，新增 `openspec/specs/unified-todo/spec.md`；项目 API 通用约束继续遵循 `openspec/specs/api-contract/spec.md`。
- 前端：新增 LogicFlow core 依赖、独立流程设计器路由与组件，调整流程定义列表、审批中心、类型和 API client。
- 后端：调整审批定义图模型、BPMN 生成器、Flowable 适配端口、实例办理与历史查询；新增统一待办模块及公开 API。
- 数据库：将当前未发布的审批迁移收敛为定义、版本、实例和统一待办四张项目表，删除重复任务、候选人和意见表设计。
- 验证：覆盖流程图结构与条件校验、BPMN 部署和分支执行、统一待办投递/完成、审批权限、前端画布交互、构建和治理检查。
