## Context

`add-archive-governance-foundation` 已经实现治理方案、版本、适用范围、绑定和默认版本解析，但管理页面只暴露方案和版本列表。后端已有 `PUT /scopes` 与 `PUT /bindings` 覆盖保存接口，缺少对应的 `GET` 读取接口，导致前端无法稳定展示和编辑既有配置。

治理方案在产品语义上是装配根，而不是简单版本字典。管理端需要围绕“当前版本采用了哪些配置、适用于哪里、默认解析命中谁”组织信息。

## Goals / Non-Goals

**Goals:**

- 让管理员在治理方案页面选中一个版本后，可以读取和维护该版本的适用范围。
- 让管理员可以读取和维护该版本绑定的本体、规则集、分类方案、著录方案和档号规则等配置引用。
- 让管理员可以输入全宗编码和分类编码，试算默认治理版本解析结果。
- 保持接口和页面实现与现有 Ant Design 企业后台风格一致。

**Non-Goals:**

- 不实现分类方案、著录方案、档号规则实体本身。
- 不把治理方案绑定目标解析成名称联查；第一版展示引用类型、目标类型、目标 ID、目标编码和排序。
- 不改写档案主数据存储，不在治理方案版本中保存题名、文号、责任者等实例值。

## Decisions

1. 新增标准子资源读取接口。

   使用 `GET /api/v1/archive-governance-scheme-versions/{versionId}/scopes` 和 `GET /api/v1/archive-governance-scheme-versions/{versionId}/bindings`。它们是治理版本下的小集合资源，数量有限，沿用 `CollectionResponse`，不引入分页。

2. 页面采用“版本工作台”而不是独立新入口。

   治理方案、版本、适用范围、绑定和默认解析属于同一管理上下文。保留现有方案和版本列表，在选中版本后追加详情 Tabs，能减少跳转和上下文丢失。

3. 绑定目标第一版保持引用展示。

   治理绑定表允许后续配置引用，但很多目标实体尚未实现。第一版不做名称联查和下拉选择，避免制造不存在的聚合能力；管理员可维护目标类型、目标 ID 或目标编码。

4. 已发布、冻结和退役版本只读。

   Service 已限制非草稿版本不能覆盖范围和绑定；前端同步禁用保存动作，避免给用户错误预期。后端仍是最终约束。

## Risks / Trade-offs

- [引用可读性不足] 只展示目标 ID 或编码不如名称友好。→ 后续对应配置实体完成后，再为绑定目标增加选择器或名称投影。
- [覆盖保存误删] 范围和绑定接口语义是整体替换。→ 前端保存前以当前表格完整行作为 payload，并保留空表代表清空配置的明确语义。
- [默认解析失败] 没有已发布默认版本时试算会返回错误。→ 前端直接展示后端错误信息，不伪造默认结果。
