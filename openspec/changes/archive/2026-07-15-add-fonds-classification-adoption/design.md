## Context

现有 `archive-metadata-routing` 规格将全宗和分类建成两个正交维度：全宗表达来源整体，分类表达字段模板和动态表路由。这个取舍保留了模板复用，但没有表达“某全宗可用哪些分类节点”的业务事实。

刚新增的治理方案工作台已经提供治理版本装配入口，`CLASSIFICATION_SCHEME` 绑定类型也已存在。当前缺少的不是把分类改成全宗子表，而是把分类方案建成部署内稳定对象，并允许全宗限定可用分类节点范围。

## Goals / Non-Goals

**Goals:**

- 建立分类方案模型，让分类节点归属于分类方案。
- 允许全宗维护可用分类节点范围，并能设置默认分类。
- 支持按全宗查询可用分类，供建档、导入和管理界面过滤选择。
- 在原型基线 DDL 中直接建立分类方案、分类方案归属和全宗可用分类范围。

**Non-Goals:**

- 不把分类节点直接挂到全宗下面。
- 不把动态表改成“全宗 + 分类”一套物理表。
- 不实现完整分类版本、分类节点历史冻结和跨版本迁移。
- 不实现同一部署内多个企业/部门并行混用的租户级分类隔离。
- 不实现分类方案和治理版本的深度联查展示，只保留可被治理绑定引用的方案 ID/编码。

## Decisions

1. 分类方案作为固定表，分类节点通过 `scheme_id` 归属方案。

   分类方案是制度包下的分类容器，分类节点仍是字段模板和动态表路由源。这样既允许多个全宗复用同一方案，也允许客户维护独立方案。

2. 全宗可用分类范围单独建表。

   使用 `am_archive_fonds_category_scope` 表保存 `fonds_code -> category_id`，并提供 `default_flag`。这样用户视角可以从全宗进入可用分类，但分类节点仍归属于分类方案，不需要复制成全宗子节点。

3. 默认分类方案作为原型初始兜底。

   基线 DDL 创建 `default_classification`，用于没有全宗分类范围时的默认分类解析；它不是为旧结构保留的兼容分支。

4. 分类编码唯一性先收敛到同一分类方案内。

   新分类在同一 `scheme_id` 下分类编码唯一。不同方案允许使用相同分类编码，但档案记录历史仍保存分类编码；后续如果要跨方案消除歧义，应同时依赖治理版本或分类方案 ID。

5. 按全宗查询可用分类是运行期便利接口。

   新增 `GET /api/v1/archive-fonds/{fondsCode}/categories` 返回该全宗范围内的启用分类；没有分类范围时返回全局默认分类方案下分类，避免初始系统不可用。

## Risks / Trade-offs

- [分类编码跨方案重复导致历史解释歧义] → 新建档案已保存治理方案版本，后续可用治理版本解释分类方案；当前阶段允许同编码跨方案复用。
- [默认回退隐藏配置缺失] → 默认分类方案只作为初始兜底；管理端仍应显式维护全宗可用分类范围。
- [分类树父子跨方案] → Service 校验父分类必须属于同一分类方案。
- [治理方案绑定目标可读性不足] → 本 change 只建立分类方案实体和全宗分类范围，治理工作台后续可把 `CLASSIFICATION_SCHEME` 目标 ID 联查为名称。
