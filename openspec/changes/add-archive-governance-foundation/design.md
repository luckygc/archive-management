## Context

当前项目已经具备档案分类动态表、`ArchiveItem` / `ArchiveVolume` 分对象、明细行表、结构化管理查询、全文投影、数据范围和基础权限等能力。继续面向档案馆、政府、企业、高校和业务系统归档扩展时，如果每个规范直接落成专用字段、专用 profile 或专用流程，后续会出现分类、著录、档号、保管期限、移交和四性检测各自维护规则的分裂。

`docs/archive-knowledge/governance-ontology-and-rules.md` 已经把候选抽象沉淀为“治理方案 + 本体 + 本地规则”三层。此 change 将其中最基础的合同迁入 OpenSpec：治理方案负责版本装配，本体负责语义解释，本地规则负责受控行为。

## Goals / Non-Goals

**Goals:**

- 建立治理方案和版本模型，让一个机构或客户采用的本体、规则和后续分类/著录/档号配置可发布、冻结和追溯。
- 建立本体核心模型，让对象类型、属性类型、关系类型和事件类型有统一语义，不把所有字段都压成动态字段。
- 建立本地规则引擎合同，用结构化条件树、事实解析、操作符白名单、effect 和决策追踪表达本地规则。
- 保持现有 `ArchiveItem`、`ArchiveVolume`、分类动态表、明细行表和全文投影的物理存储边界。
- 为后续分类方案、著录方案、档号规则、保管期限、归档移交和四性检测 change 提供稳定挂载点。

**Non-Goals:**

- 不在本 change 内实现完整分类方案版本迁移、著录模板、档号生成流水、保管期限表、正式移交流程或四性检测任务。
- 不引入 RDF、OWL、图数据库、通用三元组表或主数据 EAV 模型。
- 不引入 SpEL、MVEL、JavaScript、Groovy 或自由 SQL 作为规则语言。
- 不改变现有 archive-metadata 和 archive-record-search 的核心查询合同。
- 不内置高校、机关、企业或行业专题模板；模板后续通过独立 change 或数据初始化策略处理。

## Decisions

### 决策 1：治理方案作为装配根

采用 `ArchiveGovernanceScheme` / `ArchiveGovernanceSchemeVersion` 作为本体、规则和后续分类/著录/档号配置的装配根。档案记录保存采用的治理方案版本，用于历史解释和后续迁移。

替代方案是让分类方案直接挂规则和字段。该方案短期简单，但会把著录、档号、保管、移交、利用控制等配置全部挤进分类树，分类会变成万能聚合。治理方案单独建模能把“机构采用哪套制度”与“分类节点如何组织档案”分开。

### 决策 2：本体解释语义，不承载主数据

本体表只保存对象类型、属性类型、关系类型和事件类型定义，以及它们到固定字段、动态字段、明细字段、文件组件字段和过程对象的映射。档案主数据继续存放在 `am_archive_item`、`am_archive_volume`、分类动态表、明细行表和文件组件表。

替代方案是用 EAV 或三元组表保存所有属性值。该方案会让跨行业扩展看起来统一，但会削弱排序、唯一性、索引、权限过滤、统计和执行计划可控性，不符合当前项目 PostgreSQL + 动态 typed columns 的方向。

### 决策 3：规则引擎只返回决策

规则引擎输入事实上下文和规则定义，输出 `RuleDecision`，包括阻断、警告、建议值、派生值、访问拒绝、脱敏要求、补充检测要求等 effect。规则引擎不直接写数据库、不发布事件、不启动流程，也不拥有事务边界。

替代方案是让规则 effect 直接执行副作用。这会让事务、审计、回滚和权限边界变得隐式，后续排障困难。把副作用留给用例 Service，可以保持现有 `web -> service -> manager -> repository/mapper` 边界。

### 决策 4：规则语言使用结构化 AST

规则条件使用受控 JSON AST。字段引用必须解析到固定字段、动态字段、明细字段、文件组件字段、关系字段或上下文字段。操作符按字段类型白名单限制。effect 使用枚举和结构化参数。

替代方案是引入表达式语言或脚本引擎。该方案表达力更强，但会引入安全、性能、可审计、迁移和可视化配置问题。第一版需要的是可验证、可解释、可追踪的本地规则，而不是通用脚本平台。

### 决策 5：发布后版本冻结

治理方案版本、本体定义和规则定义发布后不可原地修改；需要变更时创建新版本。历史档案继续引用旧版本，新档案使用当前发布版本。

替代方案是允许在线编辑已发布配置。该方案对配置维护方便，但会破坏历史档案解释能力，尤其是档号、著录项、开放规则和四性检测规则发生制度性变化时无法追溯。

## Risks / Trade-offs

- [范围偏大] 治理方案、本体和规则同时引入，可能导致一次实现跨模块过多。→ 规格放在一个 foundation change，实施任务按治理方案、本体、规则、集成和前端分阶段完成。
- [抽象过度] 本体可能被误用为万能配置中心。→ 规格明确本体不保存主数据，查询不直接扫本体配置。
- [规则表达不足] 受控 AST 不如脚本灵活。→ 第一版优先覆盖校验、派生、访问、归档、移交、检测等可结构化规则；确有复杂规则时先新增 operator 或 fact resolver，不开放自由脚本。
- [历史数据迁移] 既有档案没有治理方案版本。→ 迁移时创建默认治理方案版本，并为现有记录回填默认版本或在读取时兼容空版本。
- [查询性能] 跨分类本体属性查询可能需要多个动态表。→ 第一版只要求本体属性映射和受控解析，复杂跨分类精确检索后续通过投影或专题索引扩展。

## Migration Plan

1. 新增治理方案、本体定义、规则定义和规则追踪表，不改写现有动态表数据。
2. 初始化一个默认治理方案草稿版本，用于承接现有分类、字段和规则。
3. 实现本体和规则定义的保存、校验、发布、冻结和读取接口。
4. 将新建档案、规则校验和后续配置入口接入治理方案版本。
5. 对既有档案采用默认版本回填或读取兼容策略，避免历史数据不可读。
6. 回滚时保留新表，不再让业务入口读取治理方案版本；已写入的治理配置作为未使用数据保留。

## Open Questions

- 现有档案记录迁移时，是强制回填默认治理方案版本，还是允许旧数据版本为空并在读取时解释为默认版本。
- 治理方案是否第一版就支持按全宗适用，还是先支持全局默认方案再扩展全宗绑定。
- 前端规则配置第一版是否只支持表单式条件树，还是同时提供只读 JSON 预览。
