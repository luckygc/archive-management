## Context

当前 archive-management 的档案模块只有 `am_archive_record`、`am_archive_volume`、`am_archive_volume_item` 和 `am_archive_record_storage_object` 四张基础表及对应实体。它们能表达“档案记录、案卷、卷内关联、文件关联”，但没有全宗、档案分类、字段定义、自动建表和动态表路由能力。

用户已确认首版动态表按档案分类建表，不按“分类 + 全宗”建表；字段类型只覆盖文本、整数、小数、日期、日期时间。枚举不作为物理字段类型，而是运行时动态选项。

## Goals / Non-Goals

**Goals:**

- 管理全宗，作为档案记录归属维度。
- 管理多层级档案分类，作为字段模板、动态表配置和路由入口，不再单独引入门类/分类分组。
- 按档案分类自动创建或增量维护动态数据表。
- 通过统一主表保存全宗、分类、状态和档号；题名等业务字段按分类字段定义进入动态表。
- 通过分类路由读取动态字段，支撑档案库列表。
- 修复现有档案实体和表设计中的路由字段、删除标记一致性、案卷关联一致性和业务锁问题。
- 提供 PC 端全宗管理、档案分类字段配置、建表动作和动态档案列表。

**Non-Goals:**

- 首版不支持物理删除动态列，不支持修改已建字段类型。
- 首版不实现文件、人员、组织、金额、关联档案等扩展字段类型。
- 首版不把分类绑定到全宗；同一分类模板可用于多个全宗。
- 首版不实现完整档案记录编辑流程，只提供动态列表和元数据维护闭环。

## Decisions

### 多层分类建表，主表承载全宗

档案分类只保留父子层级，不再保留门类或分类分组。动态物理表按设置了字段定义的档案分类创建，例如 `am_archive_data_c_1`。父级分类可以只承担归类，也可以在设置字段后拥有自己的物理表。全宗不移动到分类表，因为分类描述字段结构，全宗描述档案归属，两者是正交维度。统一主表 `am_archive_record` 保留 `fonds_code` 和 `fonds_name`，动态表主键 `id` 与 `am_archive_record.id` 一致，并保存分类字段值。

替代方案是动态表同时冗余 `fonds_code`，可以减少列表查询 join，但会引入主表和动态表的全宗一致性问题。当前系统仍处在单体起步阶段，优先保持数据真相源清晰。

### 字段定义驱动 DDL

字段定义表保存字段编码、名称、类型、长度、精度、是否列表显示、是否搜索、排序和启用状态。自动建表时：

- 新分类未建表时创建动态表；
- 已建表时只追加缺失列；
- 禁用字段只影响运行时展示和查询，不删除物理列；
- 已建字段类型不允许变更。

字段编码转换为列名时统一使用 `f_<field_code>`，字段编码必须限制为小写字母、数字和下划线，避免 SQL 注入和非法列名。

### 路由由分类元数据决定

档案记录写入时先创建 `am_archive_record`，再按分类找到动态表写入同 ID 动态行。查询时先基于主表过滤全宗、分类、状态和删除状态，再按分类动态表用 `d.id = r.id` 补齐字段值。`am_archive_record` 不保存动态表名或动态表行 ID，动态表名由分类元数据决定。

### 删除标记和删除审计

业务表统一使用 `deleted_flag` 表示删除状态，避免同一系统内同时出现 `deleted_flag` 和 `deleted_at` 两种业务删除口径。删除时间、删除人、删除原因和记录快照如果需要保留，写入独立删除审计表，不污染业务表结构。

### 案卷关联一致性

案卷和档案记录必须属于同一全宗和分类。数据库跨表一致性难用普通外键表达，首版在 Service 中校验 `volume.fondsCode/categoryCode` 与 `record.fondsCode/categoryCode` 一致；关联表可冗余 `fonds_code/category_code` 以便审计和查询。

### 记录级业务锁

业务锁放在 `am_archive_record` 主表上，使用 `locked_flag`、`lock_reason`、`locked_by`、`locked_at` 表达。锁定后禁止修改主表字段、动态分类字段、删除记录、加入案卷和变更附件关联；查看和解锁仍允许。业务锁解决流程或制度上“不允许动”的问题，不替代后续可能引入的乐观锁。

## Risks / Trade-offs

- 动态 DDL 注入风险 → 字段编码和表名只允许服务端生成或白名单校验，DDL 使用受控标识符拼接。
- 字段类型变更风险 → 首版禁止修改已建字段类型，只允许追加字段和修改展示配置。
- 动态列表跨分类查询复杂 → 首版档案库列表按单个分类查看；未选择分类时只展示主表通用字段。
- 主表和动态表分离带来 join 成本 → 主表先过滤再按分类补字段，避免跨多张动态表做大查询。
- 删除审计恢复复杂 → 首版先固化独立删除审计表和快照合同，恢复操作后续在该合同上实现。
- 分类多层级循环风险 → 分类更新时禁止把自身或自身子孙设置为父级，删除父级时先要求没有未删除子分类。
- 业务锁漏校验风险 → 所有档案记录写入口都在 Service 层统一查询锁状态并拒绝已锁记录，关键更新 SQL 同时带 `locked_flag = false` 条件。
