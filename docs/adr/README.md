# 架构决策记录

本目录保存需要长期解释“为什么这样设计”的架构决策记录（ADR）。当前稳定架构以 [`docs/architecture.md`](../architecture.md) 为准；ADR 记录决策形成时的背景、取舍和后果，不复制当前架构全文，也不替代 OpenSpec 业务合同。

## 何时创建

仅在出现需要长期保留理由的真实决策时创建 ADR，例如：

- 改变跨模块依赖、进程边界或持久化职责。
- 引入或替换关键基础设施、协议或数据模型。
- 在多个可行方案之间作出具有长期成本的取舍。
- 明确推翻、取代或限制已有架构决策。

局部实现细节、临时排障记录、业务需求和未作出的设想不写 ADR。

## 命名与状态

文件名使用四位递增编号和小写 kebab-case，例如：

```text
0001-use-postgresql-only.md
0002-isolate-file-preview-service.md
```

每份 ADR 至少包含：

```markdown
# ADR-NNNN：标题

- 状态：提议 | 已接受 | 已取代 | 已拒绝
- 日期：YYYY-MM-DD

## 背景

## 决策

## 替代方案

## 后果
```

若新 ADR 取代旧决策，应在两份文件中互相链接；历史 ADR 不删除或改写成新结论。

## 与其他真相源的关系

- 业务、权限、状态机和验收场景写入 `openspec/specs/`，不写入 ADR。
- 已稳定生效的包边界和组件职责同步到 `docs/architecture.md`。
- 领域词汇使用根 [`CONTEXT.md`](../../CONTEXT.md)。
