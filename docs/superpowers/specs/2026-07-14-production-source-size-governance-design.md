# 生产源码职责收敛设计

## 背景

前一轮代码精简已经删除部分总门面、拆分前端 API 与类型，并固化 Spring Bean 边界，但生产源码仍有少数超长文件。当前最突出的后端文件是 `ArchiveItemRoutingService` 和 `ArchiveMetadataService`，前端则有多个接近六百有效代码行的路由页面。

本轮把有效代码行作为职责过载的检查信号：前端生产业务文件超过 300 行进入软提示、超过 500 行硬失败，后端生产业务文件超过 500 行进入软提示、超过 700 行硬失败。有效代码行不包括空白和注释。测试、数据库迁移、Mapper XML、生成代码和第三方代码不在阈值范围内。

行数不是机械验收规则。文件只有在形成真实业务边界、减少依赖或集中副作用时才拆分；不为了达标新增空壳转发类、参数矩阵 helper、接口、工厂或兼容层。

## 目标

- 将生产业务文件收敛到前端 300–500 行、后端 500–700 行的合理区间，超过软提示线时复核职责，超过硬失败线时阻断检查。
- 让每个拆出的类、组件或 composable 对应可命名、可独立理解的业务职责。
- 保持现有 HTTP API、数据库结构、权限、事务、错误响应和前端交互行为不变。
- 复用已经存在的专项 Service、组件与 composable，不建立新的通用 CRUD 框架。
- 用现有行为测试、类型检查和架构测试证明重构前后行为稳定。

## 范围

### 纳入

- `server/src/main/java` 下项目自有生产业务代码。
- `web/src` 与 `frontend-core/src` 下项目自有生产 TypeScript、Vue 和 CSS 代码。
- `preview` 下项目自有 Go 生产代码；当前没有明显超过目标的文件，仅纳入最终复核。

### 排除

- 所有测试源码。
- Flyway 迁移、样例 SQL 和第三方框架 DDL。
- MyBatis Mapper XML；其拆分应由查询边界和持久化策略决定，不由行数单独驱动。
- 构建产物、生成代码、技能资产和依赖代码。

## 方案选择

### 采用：按稳定业务职责增量拆分

后端沿现有 `web -> service -> manager -> repository/mapper` 方向，把查询、写入、元数据子域和权限协作迁入具体 Service。前端以完整业务区域为单位提取组件或 composable，路由页面继续负责跨区域选择、刷新和权限编排。

该方案既能降低单文件认知负担，也能让依赖和副作用边界更清楚。每一批都保持编译和测试通过，不长期维护新旧双入口。

### 不采用：按固定行数机械切割

机械切割容易产生只转发调用的类、超长参数列表和难以追踪的状态共享。即使文件行数下降，也不会降低整体复杂度。

### 不采用：一次性重写领域模型

一次性调整实体、接口、持久化和页面状态会扩大回归范围。本轮只移动已存在的职责，不改变业务合同或数据模型。

## 后端设计

### 档案记录

`ArchiveItemRoutingService` 当前仍包含管理查询、全文发现、回收站、游标分页、动态查询条件、档案写入、动态字段转换、权限和审计。拆分边界如下：

- 查询 Service：列表、搜索、发现、回收站、排序、游标及动态查询条件构造。
- 写入 Service：创建、修改、删除、详情写模型装配、动态字段写入和操作审计。
- 已有权限、电子文件、卷件关系和搜索投影 Service 继续承担原职责，不新增平行能力。
- 查询使用的动态表和动态字段 SQL 继续由 MyBatis 持有；固定实体 CRUD 继续由现有 Jakarta Data Repository 持有。

Controller 和跨模块调用直接依赖真实用例 Service。迁移完成后删除旧入口，不保留兼容门面。仅在调用方数量和构造器复杂度确有必要时保留一个有业务含义的协调 Service。

### 档案元数据

`ArchiveMetadataService` 保留字段定义、有效布局、动态表构建和唯一约束的事务协调职责，底层校验与技术操作继续复用现有专项 Service。全宗、分类方案、门类及全宗门类范围按现有实体边界形成具体 Service。固定表继续使用 Jakarta Data Repository，不把固定 CRUD 迁入 MyBatis，也不把动态 DDL 迁入 Repository。

公共映射和校验默认留在唯一拥有该业务不变量的 Service 内。只有存在真实重复且语义一致时才提取协作类。

### 其余后端热点

在前两个文件完成并稳定后，依次检查 `ArchiveDataScopeService`、`ArchiveOntologyService` 和 `ArchiveLocalRuleService`。只有能识别独立权限边界、事务边界或业务子域时才拆；接近阈值但职责单一的文件允许合理浮动。

本体对象、属性与映射由 `ArchiveOntologyService` 负责，关系与事件由 `ArchiveOntologyRelationService` 负责；Controller 直接依赖两个真实用例 Service，不在前者保留纯转发入口。

## 前端设计

路由页面保留以下职责：

- 路由参数和页面级权限。
- 当前主对象选择。
- 跨业务区域刷新和联动。
- 页面级 Loading、Empty 和不可恢复错误状态。

以下内容可按完整业务区域提取：

- 筛选与结果表格。
- 创建或编辑 Drawer/Dialog 及其表单状态。
- 详情、版本、适用范围、绑定关系等独立面板。
- 只服务一个业务区域的请求编排和状态，放入同目录 composable。

子组件通过语义明确的 props、事件和 `defineModel` 协作，不直接读取路由或页面 Store。不得为了缩短页面把每个表单区块拆成只转发十几个属性的小组件。继续使用 Element Plus 原生组件和现有设计 token，不改变视觉层级和交互文案。

优先处理当前超过 500 行硬阈值的页面。其他超过 300 行软提示线的页面只有在形成稳定业务区域时才随批次处理。

## 数据流、事务与错误

- 后端写入事务仍位于实际完成原子状态变更的具体 Service public 方法。
- 同一 Spring Bean 的 public 方法不互调，共享实现使用 private 方法或另一个具有独立业务边界的具体 Bean。
- 权限和数据范围校验不能下沉到前端，也不能在拆分时遗漏。
- HTTP 请求、响应、ProblemDetail、状态码和错误文案保持不变。
- 前端成功后的列表刷新、选中项同步和弹层关闭行为保持不变。
- 前端继续使用 `frontend-core` 的统一错误解析，不新增第二套状态源。

## 测试策略

重构不新增业务行为，以现有行为测试作为保护网。每批步骤如下：

1. 运行与目标文件对应的测试，确认基线通过。
2. 增加能约束新边界的架构测试或聚焦测试，并确认它在迁移前按预期失败；若现有测试已完整覆盖行为，不为私有实现编写脆弱测试。
3. 迁移一个完整业务用例及全部调用方，删除旧入口。
4. 运行聚焦测试、编译和格式化检查。
5. 重新统计有效代码行并检查新文件是否只是转发层。

## 实施顺序

1. 建立可重复的生产源码有效行统计，并记录初始热点。
2. 拆分 `ArchiveItemRoutingService` 的查询和写入职责。
3. 拆分 `ArchiveMetadataService`，复用现有专项 Service。
4. 处理仍超过 700 行硬阈值或超过 500 行且职责确实混杂的后端业务文件。
5. 按完整业务区域拆分前端热点页面。
6. 完整验证并复核全部生产源码。

## 完成标准

- 后端生产业务文件超过 500 行时提示复核，超过 700 行时检查失败。
- 前端生产业务文件超过 300 行时提示复核，超过 500 行时检查失败。
- 合理例外必须职责单一，且拆分会增加参数转发、状态同步或持久化边界混乱。
- 不存在为行数达标新增的空壳转发类、无收益接口或通用 CRUD 抽象。
- HTTP、数据库、权限、事务、错误和前端交互合同保持稳定。
- 后端完整测试以及前端检查、测试和构建通过。
