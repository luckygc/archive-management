# 档案管理系统全量收口设计

## 目标

在不因 Beta、Milestone 或预发布标签降级依赖的前提下，修复现有 MVP 审查发现的产品闭环、架构接入、API 合同、交付治理和文档问题，使系统从“后端能力较完整的内部验证版”演进为可供早期用户持续使用、可部署和可验收的档案管理系统。

本设计是跨子项目的总设计。实施时拆分为六个独立 OpenSpec 变更，每个变更都必须形成可运行、可测试、可单独审查的交付物，不用一个超大变更同时修改所有模块。

## 全局约束

- 依赖版本只允许升级，不因 Beta、Milestone、RC 或预发布标签降级、替换或回退依赖。
- 新增依赖必须解决已经出现的需求；已有框架能够完成的能力不新增平行基础设施。
- 后端继续采用 Spring Boot 模块化单体，业务模块不得直接依赖 Flowable、S3 SDK 或预览服务原生合同。
- Web 继续采用 Vue 3、TypeScript 和 Element Plus，不新增第二套组件体系或通用 CRUD 框架。
- HTTP API、DTO、分页、错误合同和 ID 类型遵守 `openspec/specs/api-contract/spec.md`。
- 固定表继续使用 Jakarta Data Repository；动态表、复杂查询、批处理和报表继续使用 MyBatis。
- 所有业务写入由服务端权限、数据范围、状态约束和数据库约束共同守护，前端权限只用于改善可见性和操作体验。
- 不主动启动 Web 开发服务器；前端验证使用静态检查、组件测试、构建和浏览器自动化测试。

## 变更分解与顺序

### 1. `close-archive-pc-mvp`

这是第一优先级变更，先关闭现有档案能力从后端到 PC 的使用缺口。

#### 游标分页

- 档案搜索和档案管理统一复用现有 `CursorPagination`。
- `useArchiveItemSearch` 保存已提交查询、排序、页大小、当前游标和前后游标，不把草稿筛选条件用于翻页。
- 首次查询、筛选提交、排序变化和页大小变化必须清除旧游标；上一页、下一页和刷新必须继续使用已提交查询。
- 页面必须覆盖无下一页、加载中、游标失效和请求失败状态。
- 使用超过 100 条且排序字段有重复值的数据验证无重复、无遗漏翻页。

#### 档案生命周期与字段

- 档案管理列表提供查看、编辑、锁定、解锁、删除、电子文件和审计操作。
- 锁定需要填写原因；解锁和删除需要二次确认；操作成功后刷新当前已提交查询，不重置筛选条件。
- 编辑器完整承载物理字段、密级、保管期限和现有动态字段。详情模式只读展示同一合同，不再提交固定空的 `physicalFields`。
- 服务端继续作为状态转换和权限真相源；前端根据记录状态禁用无效动作，并展示服务端返回的可执行错误原因。

#### 案卷、关系与明细行

- 新增案卷管理页面，支持按全宗和分类筛选、创建案卷、查看案卷、将档案加入案卷和查看案卷内档案。
- 档案详情增加关系页签，支持查看、创建和删除有明确关系类型的档案关联。
- 分类配置增加明细表定义和字段维护；档案编辑器根据已构建的明细表维护行数据。
- 不新增通用低代码表格设计器；字段类型、校验和渲染继续复用现有元数据类型。

#### 权限与错误状态

- 路由元数据增加所需权限码；菜单、分组、页签和直接路由访问使用同一判断函数。
- `superAdmin` 明确视为拥有全部权限；普通用户以服务端返回的权限码集合为准。
- 无任何可见子菜单的分组不显示；直接访问无权路由进入统一的无权限页面，不依靠接口失败表达导航权限。
- 关键列表使用页面内错误状态和重试动作保留失败上下文；瞬时操作结果继续使用 `ElMessage`。

#### 真实工作台

- 删除所有硬编码统计和待办。
- 新增用户数据范围内的工作台摘要 API，返回可访问档案数、草稿数、锁定数和电子文件数。
- 审批完成前，工作台不伪造待办；审批变更上线后再增加真实待办摘要。
- 摘要查询失败时单独显示可重试错误，不阻断其他工作区导航。

### 2. `complete-archive-intake`

归档接收从入口占位发展为可恢复的批次处理闭环。

- 一个接收批次绑定目标全宗、档案分类、提交人和来源说明。
- 批次状态固定为草稿、校验中、待修正、可提交、提交中、已完成、已失败和已取消；状态转换由服务端控制。
- 首版数据来源为上传项目标准导入模板，不同时开发外部数据库、接口同步或消息订阅适配。
- 上传后先写暂存记录，执行字段映射、必填、类型、唯一规则和数据范围校验；错误按行号、字段和稳定错误码返回。
- 用户可下载错误清单、替换源文件后重新校验，并在全部通过后显式提交。
- 提交以幂等批次动作创建档案；部分写入不得暴露为完成状态，失败后保留批次和诊断信息供重试。
- Web 页面包含批次列表、创建向导、校验结果和批次详情，不保留“入口已预留”文案。

### 3. `complete-approval-workflow`

继续现有 `add-flowable-approval-workflow`，完成剩余任务并以归档发布作为首个真实业务接入。

- Flowable 仅作为 BPMN 运行时，所有项目业务合同位于审批模块；其他业务模块只能调用审批模块公开 Service。
- 首版流程定义只支持开始、顺序用户任务和结束，候选人只支持指定用户，不引入角色表达式、会签、并行网关或脚本节点。
- 完成流程定义草稿、版本发布、实例、任务快照、审批意见和业务绑定项目表。
- 完成我的待办、我发起的、已办理、实例详情、同意、驳回、撤回和终止 API 与页面。
- 首个业务类型固定为归档发布：档案从草稿发起审批，通过后进入已归档；驳回回到可编辑草稿；流程中档案不可删除或修改关键字段。
- 审批项目表和 Flowable 操作处于同一事务边界；异常时不提交项目状态，重复办理返回稳定冲突错误。
- 工作台在该变更完成后读取真实待办数量和近期事项。

### 4. `integrate-file-preview`

保留独立 Go 预览服务，但把它接入主系统安全边界和电子文件体验。

- 浏览器只调用 Spring Boot 项目 API，不直接访问 Go 服务。
- Spring Boot 根据当前用户、档案数据范围和电子文件下载权限打开文件，再调用预览服务。
- 基础设施层提供预览客户端和配置；业务模块通过明确的文件预览 Service 使用，不依赖 Go 服务 DTO。
- 首版同步预览限制输入大小和请求超时，支持 PDF、图片和文本；不支持的 Office、CAD、音视频返回可下载而不可预览的明确状态。
- 前端电子文件页签提供预览动作，在 Drawer 内展示加载、不可预览、转换失败和下载替代动作。
- 预览失败不改变原文件、不影响下载，也不生成持久化派生对象。
- Go 加入 `mise.toml` 管理的工具链，根任务统一使用 `task preview-test`、`task preview-build` 和 `task preview-run`；文档不再出现不存在的 Makefile 命令。

### 5. `complete-system-settings`

替换存储配置和系统参数占位页，但不构建危险的任意配置编辑器。

#### 存储配置

- 页面只展示当前 provider、endpoint 脱敏信息、bucket、连接状态和最近检查时间。
- 后端提供读取状态与显式连接检查 custom method；响应不得包含 access key、secret key、会话令牌或完整签名 URL。
- 运行时凭据仍由部署配置管理，不允许 Web 写入。

#### 系统参数

- 只维护项目明确声明可运行时修改的业务参数白名单，首批包括导入单批最大行数、电子文件单文件大小上限和同步导出最大行数。
- 参数记录包含键、类型、值、说明、更新时间、更新人和版本；后端按白名单进行类型、范围和并发版本校验。
- Spring Boot、数据库、缓存、会话、Flowable 和 S3 凭据不进入该参数表。
- 页面支持列表、编辑和恢复单个参数默认值，不支持任意键创建。

### 6. `align-api-and-delivery`

在功能闭环稳定后整理合同和交付入口，避免与大功能开发交叉重写。

#### API 合同

- HTTP 请求和响应对象使用动作或视图语义明确的 `Request`、`Response` 名称；只服务 HTTP 的对象位于 `web` 包。
- 删除 `SearchArchiveItemsRequest` 中由统一分页解析器负责的 `limit`、`cursor` 等遗留字段和重复 Service 重载。
- 所有可增长集合必须能通过公开请求参数继续翻页；不能返回客户端无法使用的游标。
- 导出统一使用 `POST /api/v1/archive-items:export` 请求体。小规模同步导出返回受权限保护的下载链接；超过同步阈值时再由独立 OpenSpec 引入异步任务，不在本次预留空壳。
- 整理过程中保持 ProblemDetail、数字 ID 和现有 URL 版本规则不变。

#### 规格、CI 与验收

- 将已经完成且验证通过的 OpenSpec 变更归档进正式规格；未完成审批变更在完成前保持活动状态。
- 更新 API、架构、部署、开发、运维和 MVP 差距文档，使其与代码和任务命令一致。
- 增加 CI，执行依赖安装、`pnpm check`、`pnpm test`、前端构建、后端测试、OpenSpec 严格校验、源码职责检查以及 Go 预览测试和构建。
- 增加浏览器级 MVP 冒烟测试，覆盖登录、权限菜单、分类选择、创建档案、超过 100 条游标翻页、锁定解锁、电子文件、案卷和审计。
- Intake、审批和预览各自增加独立端到端场景，不用单个巨型测试串联所有模块。

## 总体数据流

```text
Vue PC
  │
  ▼
Spring Boot 项目 API
  ├─ 认证、权限、数据范围、ProblemDetail
  ├─ 档案 / 接收 / 审批 / 系统参数业务模块
  ├─ Jakarta Data：固定 CRUD 表
  ├─ MyBatis：动态表、复杂查询、批次和统计
  ├─ Flowable 基础设施适配：只服务审批模块
  ├─ S3 文件存储：通过 FileStorageService
  └─ 预览客户端：鉴权后调用 Go Preview
          │
          ▼
       Go Preview
```

Web 不持有第二份权限或业务状态真相。浏览器中的菜单可见性、按钮禁用和表单校验用于提前反馈，所有敏感读取、写入、状态转换和文件操作仍由 Spring Boot 重新校验。

## 错误与恢复设计

- 列表加载失败保留已提交筛选和排序，页面提供原位重试。
- 写操作失败保留用户输入；字段错误使用 `fieldViolations` 回填，冲突和前置条件错误显示可执行原因。
- 游标失效时提示数据已变化，并从相同已提交查询的第一页重新开始。
- Intake、审批办理和连接检查使用幂等动作或版本条件避免重复提交。
- 外部预览服务失败只降级预览，不伪造成功、不吞掉诊断，也不影响原文件下载。
- 所有服务端错误保留 `traceId`，前端不解析异常类名或自由文本判断业务分支。

## 测试策略

- 每个功能和缺陷修复遵循红、绿、重构：先写会因缺失行为而失败的测试，确认失败原因，再实现最小代码。
- 前端组合式函数、权限判断、API 请求和关键页面状态使用 Vitest；页面测试验证真实按钮、禁用、错误和分页交互。
- 后端 Service 测试覆盖权限、数据范围、状态机、幂等和失败恢复；Controller 测试覆盖 URL、请求响应和 ProblemDetail；数据库集成测试覆盖 PostgreSQL 约束、动态 SQL 和游标稳定性。
- ArchUnit 继续验证模块依赖、HTTP 映射、Repository 操作注解、Flowable 隔离和基础设施边界。
- 浏览器 E2E 使用专用测试数据和两个不同权限用户，验证界面可见性与服务端拒绝同时成立。
- 每个子项目完成时运行其定向测试和全量相关验证；总目标完成前运行完整前端、后端、OpenSpec、Go、构建和 E2E 验证。

## 完成标准

只有以下条件全部满足，才能认定本总目标完成：

- 六个子项目均有已验证的 OpenSpec 合同和实现，任务全部完成。
- 审查中发现的硬编码数据、占位页面、无法继续的游标、后端独占功能、权限菜单缺口、预览未集成和文档命令漂移全部消失。
- 依赖没有因预发布标签被降级；如有调整，只能是升级或为真实需求新增依赖。
- 用户可以通过 PC 完成档案配置、创建与编辑、分页搜索、锁定解锁、案卷关系、明细行、文件预览与下载、归档接收、归档发布审批、审计和受限系统配置。
- 全量验证通过，工作区中的规格、文档、代码和自动化命令相互一致。
