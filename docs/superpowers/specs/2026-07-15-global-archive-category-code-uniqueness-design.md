# 档案分类编码全局唯一设计

## 背景

归档 `add-fonds-classification-adoption` 后，长期规格与运行时模型出现冲突：分类方案规格允许不同方案复用分类编码，数据库也只约束 `(scheme_id, category_code)`；但动态表名仅由分类编码、item/volume 对象类型和字段域生成，档案读取与全文投影也按分类编码解析分类。若两个方案使用相同编码，系统可能路由到同一物理表或解析到错误分类。

项目尚未正式发布，不需要保留方案内重复编码的兼容分支。为保持动态表稳定业务键、现有查询入口和运行时结构不变，分类编码收敛为跨分类方案全局唯一。

## 目标与非目标

目标：

- 未逻辑删除的档案分类使用全局唯一 `category_code`。
- 创建和修改分类时提前返回可理解的统一错误。
- 数据库约束负责并发写入下的最终一致性。
- `archive-classification-scheme` 与 `archive-metadata` 对唯一性采用同一合同。

非目标：

- 不把 `scheme_id` 加入动态表名、档案主表或全文投影路由。
- 不重命名现有动态表，不增加迁移映射或兼容查询。
- 不改变分类方案、父子分类同方案、全宗可用分类范围等既有能力。
- 不顺带调整 Repository 与审计统一任务中的其他代码。

## 设计

### 业务不变量

分类方案继续负责组织分类节点和字段模板，但分类编码是跨方案稳定业务键。任意两条 `deleted_flag=false` 的分类不得使用相同 `category_code`；已逻辑删除分类不占用编码。

这一选择与现有动态表命名保持一致：`category_code + item/volume + METADATA/PHYSICAL` 仍可唯一定位物理表，无需让查询、档案记录或投影额外携带方案身份。

### 持久化约束

项目未发布，直接维护当前目标结构：将分类表的活动唯一索引从 `(scheme_id, category_code)` 调整为 `category_code`，仍使用 `where deleted_flag = false` 释放逻辑删除记录占用的编码。

索引名称同步表达真实语义，不保留旧的方案内唯一索引或双重约束。

### Service 与 Repository

`ArchiveCategoryDataRepository` 增加一个明确操作注解标注的按分类编码查询方法。该方法是创建、修改分类所需的真实持久化能力，不新增通用 Repository helper。

`ArchiveCategoryService` 在 trim 并校验非空后检查全局编码冲突：

- 创建时，只要存在未删除同编码分类就拒绝。
- 修改时，允许当前分类保留原编码；若同编码属于其他分类则拒绝。
- 数据库唯一索引继续处理并发请求在提前检查之后同时写入的竞争窗口。

提前检查和数据库唯一冲突都转换为 `409 Conflict` 的项目统一 ProblemDetail，错误码使用现有 `ALREADY_EXISTS` 映射，不向客户端暴露 SQL、索引名或 Hibernate 异常细节。

### 规格校准

`archive-classification-scheme` 的“分类节点归属分类方案”改为分类编码在全部未删除分类中唯一，同时保留父子分类必须同方案。`archive-metadata` 继续采用相同全局唯一合同。

归档中的历史 delta 保持不变，用于记录当时设计；当前 `openspec/specs/` 是长期合同真相源。

## 失败路径

- 空白编码继续按现有参数校验拒绝。
- 创建重复编码返回 `409 Conflict`，并说明分类编码已存在。
- 修改为其他分类已占用的编码返回同类错误。
- 并发创建或修改绕过提前检查时，由部分唯一索引拒绝，Service 将其转换为相同的 `ALREADY_EXISTS` ProblemDetail。
- 已逻辑删除分类不参与冲突，可由新分类重新使用其编码。

## 测试与验证

- Service 单元测试覆盖创建重复编码、修改为其他分类编码、修改时保留自身编码。
- 持久化或应用集成测试确认活动分类编码全局唯一，并确认逻辑删除后可以复用。
- OpenSpec 严格校验确认两个长期规格不再互相矛盾。
- 运行后端格式检查、编译和相关测试；最终运行后端全量测试。
- 使用模式断言确认当前迁移中不再存在 `uk_am_archive_category_scheme_code_active`，长期规格不再声明“同一分类方案下唯一”。

## 回滚边界

本变更不迁移生产数据。若实施验证失败，回滚本次规格、Service、Repository、测试和目标迁移修改即可；不需要动态表重命名或数据回填。
