# 常用档案分类

本页整理常见档案门类和分类方案，用于项目建模前的业务语言统一。分类方案必须按客户或机构实际确认；这里不是默认建表清单。

## 分类层级

档案分类经常同时包含几类维度，系统建模时不能混成一个字段：

| 维度      | 含义                                       | 示例                                         | 项目建议                                       |
| --------- | ------------------------------------------ | -------------------------------------------- | ---------------------------------------------- |
| 门类      | 按档案形成领域、载体或管理制度划分的一级类 | 文书、科技、会计、照片、录音、录像、业务数据 | 可作为 `archive category` 的上位维度或分类模板 |
| 子门类    | 在某个门类下继续细分                       | 科技下分科研、基建、设备                     | 用树结构表达，不要写死到枚举                   |
| 业务分类  | 按客户业务或系统模块划分                   | 合同、报销、采购、诉讼、项目                 | 可以映射到动态表和字段集合                     |
| 保管期限  | 按保存价值划分                             | 永久、30 年、10 年                           | 是档号和归档规则常用组成项，但不是分类树本身   |
| 组织/问题 | 按形成部门或问题主题分类                   | 办公室、人事、党建、财务                     | 可参与档号生成和权限过滤                       |
| 载体/格式 | 按记录形态划分                             | 纸质、电子、照片、音频、视频、实物           | 影响文件对象、预览、长期保存和移交包           |

## 机关常见门类

机关档案常见门类可以按以下方向理解：

| 门类         | 常用代码 | 典型内容                           | 建模提示                                      |
| ------------ | -------- | ---------------------------------- | --------------------------------------------- |
| 文书         | WS       | 党务、行政管理、综合事务形成的文件 | 常按件整理，常与年度、机构/问题、保管期限组合 |
| 科技         | KJ       | 科研、基建、设备等科技活动材料     | 通常还要分科研、基建、设备等子门类            |
| 人事         | RS       | 人事管理活动形成的材料             | 权限敏感，不宜和普通文书同权限                |
| 会计         | KU / CK  | 会计核算、报表、凭证、账簿等       | 需要专题字段、保管期限和凭证关系              |
| 专业         | ZY       | 行业特有职责形成的档案             | 不同客户差异最大，应配置化                    |
| 照片         | ZP       | 照片档案                           | 常按组、张、拍摄时间和活动整理                |
| 录音         | LY       | 录音档案                           | 需要时长、格式、采集时间等技术元数据          |
| 录像         | LX       | 视频档案                           | 需要编码、时长、分辨率、预览和转码策略        |
| 业务数据     | SJ       | 业务系统数据归档                   | 需要数据结构、来源系统和可读性说明            |
| 公务电子邮件 | YJ       | 邮件归档                           | 需要发件人、收件人、主题、时间和附件关系      |
| 网页信息     | WY       | 网站或网页归档                     | 需要采集时间、URL、快照和哈希                 |
| 社交媒体     | MT       | 新媒体账号内容                     | 需要平台、账号、发布时间和采集证据            |
| 实物         | SW       | 奖牌、印章、题词、纪念品等         | 需要实物位置、尺寸、照片和利用控制            |

来源提示：`机关档案管理规定` 和多地“三合一制度”样例都采用门类、保管期限、归档范围、档号结构结合的方式。项目不应简单套用某个机关的门类表，应把门类作为可配置方案。

## 高校常见门类

高校公开分类方案通常以《高等学校档案实体分类法》为基础。常见一级类目包括：

| 代码 | 门类               | 典型内容                                       |
| ---- | ------------------ | ---------------------------------------------- |
| DQ   | 党群               | 党委、工会、团委、民主党派等党群组织形成的材料 |
| XZ   | 行政               | 行政管理部门形成的综合管理材料                 |
| JX   | 教学               | 教学管理、教学实践、学籍、成绩、培养过程材料   |
| KY   | 科学研究           | 科研管理、课题、成果、奖励、实验记录等         |
| CP   | 产品生产与科技开发 | 产品生产、技术开发、成果转化活动材料           |
| JJ   | 基本建设           | 建筑工程、规划、设计、施工、竣工、图纸等       |
| SB   | 仪器设备           | 设备购置、验收、安装、使用、维护、报废等       |
| CB   | 出版               | 出版管理、出版活动和出版物相关材料             |
| WS   | 外事               | 国际交流、合作办学、外宾来访、出国出访等       |
| CK   | 财会               | 会计核算、财务管理、凭证、账簿、报表等         |

不少学校会在上述基础上扩展学生、声像、实物、人物、校史、捐赠等门类。扩展时应坚持两个原则：

- 扩展门类要解决真实整理、权限、保管或检索问题。
- 不要因为一个门类有特殊字段，就把所有分类都拆成独立代码路径。

## 企业和业务系统常见分类

企业或业务系统归档常见分类不一定使用高校或机关门类，可按业务活动划分：

| 业务域 | 常见分类                                   | 建模提示                                   |
| ------ | ------------------------------------------ | ------------------------------------------ |
| 合同   | 合同正文、审批、补充协议、履约、变更、终止 | 需要合同号、相对方、金额、期限、项目关系   |
| 财务   | 报销单、发票、付款、凭证、账簿、报表       | 需要会计期间、金额、科目、凭证号和附件关系 |
| 采购   | 需求、询价、招标、评审、订单、验收         | 需要采购项目、供应商、合同和验收关系       |
| 项目   | 立项、计划、过程、验收、成果、变更         | 需要项目号、阶段、任务和成果文件关系       |
| 人事   | 入职、合同、考核、培训、奖惩、离职         | 权限和保管期限通常更严格                   |
| 法务   | 案件、证据、文书、裁判、执行、法律意见     | 需要案件号、当事人、证据链和敏感权限       |
| 研发   | 需求、设计、实验、测试、专利、成果         | 需要版本、项目、成果和知识产权关系         |

## 对当前项目的建模建议

1. `archive category` 不应只等同“门类”。它可能是门类、业务分类或动态表路由，需要在 OpenSpec 中固定语义。
2. 分类方案应支持多层级、代码、显示名称、适用范围、启停状态和排序。
3. 档案分类和全宗是两个维度：全宗表达来源整体，分类表达整理和检索结构。
4. 保管期限、密级、载体、年度、机构/问题等应作为分类规则或档号规则的组成项，不要都塞进分类树。
5. 支持项目级分类模板，但不要把高校分类、机关分类和企业分类写成一套硬编码枚举。
6. 分类变更要考虑已归档档案：历史记录应保留当时分类代码和档号，不应被新分类重算破坏。

## 推荐数据概念

| 概念               | 说明                                             |
| ------------------ | ------------------------------------------------ |
| 分类方案           | 某个机构或客户采用的一套分类规则                 |
| 分类节点           | 方案下的树节点，含代码、名称、父节点、层级和说明 |
| 分类版本           | 分类方案发生制度性变更时的版本                   |
| 分类适用范围       | 限定方案适用于机关、高校、企业、会计、项目等场景 |
| 分类到动态表映射   | 分类节点和动态档案表、字段集合的关系             |
| 分类到档号规则映射 | 分类节点采用的档号模板和编号流水规则             |

## 待进入 OpenSpec 的决策

- 当前项目默认分类方案是通用模板，还是每个客户/机构独立维护。
- 分类节点是否允许跨全宗复用。
- 分类代码变更后历史档号是否冻结。
- 分类与动态表是一对一、一对多，还是多分类共享同一表。
- 是否内置高校分类模板和机关门类模板。
