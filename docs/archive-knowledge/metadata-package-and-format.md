# 元数据、封装与长期保存格式

本页关注电子档案元数据、封装包和长期保存格式，主要参考 `DA/T 46-2009`、`DA/T 47-2009`、`DA/T 48-2009`、`DA/T 58-2014`、`GB/T 18894-2016` 等。

## 元数据理解

档案元数据不只是页面上的动态字段。更完整地看，至少有几类：

- **描述元数据**：题名、责任者、形成时间、档号、分类、全宗、保管期限等。
- **结构元数据**：案卷和卷内、件和附件、文件组件、关联档案、明细行关系。
- **管理元数据**：归档、移交、接收、保管、开放、销毁等管理过程。
- **技术元数据**：文件格式、大小、摘要、编码、创建软件、存储位置、版本。
- **权限和利用元数据**：密级、开放控制、访问范围、利用记录、下载记录。
- **保存元数据**：格式迁移、校验、备份、修复、长期保存策略。

当前项目的 `archive-metadata` 更偏描述元数据和动态字段管理；后续做移交、单套管理和证据效力时，需要补充管理、技术和保存元数据。

## 封装理解

电子文件封装通常用于把文件、元数据和关系打包成可交换、可校验、可接收的结构。项目上不应把封装简单理解成 zip：

- 包内需要文件清单。
- 每个文件需要摘要、格式、大小和标识。
- 元数据需要能指向对应文件或组件。
- 包结构需要表达案卷、件、附件、关联和层级。
- 接收方需要能校验数量、摘要、必填字段和结构一致性。

## 长期保存格式理解

长期保存关注的是多年以后还能不能读、解释和证明。项目上需要区分：

- 当前业务系统生成的原始文件。
- 归档时保存的版式文件或长期保存格式。
- OCR、全文抽取、预览图等派生内容。
- 文件格式迁移后的新版本和旧版本关系。

对象存储解决的是“存得下”和“取得到”，不自动解决长期保存格式、格式迁移、可信校验和可解释性。

## 对当前模型的影响

| 项目对象     | 行业规范启发                                         |
| ------------ | ---------------------------------------------------- |
| 档案分类字段 | 只表达一部分描述元数据，不要承载所有管理和技术元数据 |
| 档案条目     | 应能关联文件对象、案卷、明细行、关联档案和过程记录   |
| 文件对象     | 需要摘要、大小、格式、存储位置、版本和来源           |
| 导入导出     | 应区分普通 Excel 导入、正式归档包、移交包和证据包    |
| 全文索引     | 是派生检索能力，不是长期保存原始依据                 |
| 字段布局     | 是前端展示和录入体验，不应替代元数据语义定义         |

## 建议进入 OpenSpec 的能力

- 文件对象和档案条目的关系模型。
- 元数据字段类型、必填规则、来源和校验规则。
- 正式归档包、移交包或导出包的包结构。
- 文件摘要、格式、大小和版本的校验要求。
- 格式迁移和派生内容的关系约束。

## 暂不直接采用

- 不在当前阶段自研完整 XML 封装协议，除非明确要和外部档案馆或行业系统交换。
- 不把所有 `DA/T 46` 元数据元素一次性做成固定字段。
- 不把 OCR 文本或全文索引当成原始档案内容。

## 待确认

- 是否需要支持标准化归档包或移交包导入导出。
- 是否要为文件对象保存 MIME、扩展名、格式标识、摘要算法和版本。
- 是否需要长期保存格式转换，例如办公文档转版式文件。
- 是否需要保存 OCR、预览、缩略图、全文抽取等派生对象的来源链路。
