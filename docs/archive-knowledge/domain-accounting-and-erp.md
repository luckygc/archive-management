# 会计与 ERP 专题

本页关注电子会计档案、财务报销电子凭证和企业 ERP 系统电子文件归档，主要参考 `DA/T 94-2022`、`DA/T 95-2022`、`DA/T 104-2024` 等。

## 为什么单独成专题

会计、财务报销和 ERP 归档比普通文书档案更依赖业务系统数据。它们通常不仅有文件，还包括：

- 凭证、单据、附件、影像、审批流和业务流水。
- 来源系统、业务单号、组织、人员、金额、期间、科目和状态。
- 原始电子凭证、版式文件、结构化数据和外部凭证之间的关系。
- 审批、报销、入账、冲销、更正、归档和利用链路。

因此专题能力不适合只用“附件上传 + 动态字段”概括。

## 电子会计档案理解

电子会计档案强调会计资料形成、归集、整理、归档、保管和利用的一致性。项目上需要关注：

- 会计凭证、账簿、报表、附件和审批材料的关系。
- 结构化数据和版式文件、影像文件之间的对应关系。
- 会计期间、单位、凭证号、业务单号、金额、科目等检索维度。
- 保管期限、利用控制、审计记录和导出要求。

## 财务报销电子凭证理解

财务报销电子凭证归档更强调来源系统、凭证真实性、流程完整性和附件对应关系。项目上需要关注：

- 报销单、发票、审批记录、支付记录、入账凭证之间的链路。
- 同一个业务事项下多文件、多附件、多明细的组合关系。
- 电子凭证和纸质补充材料并存时的对应关系。
- 问题凭证、重复凭证和缺失附件的校验。

## ERP 归档理解

ERP 系统电子文件归档通常覆盖采购、销售、库存、生产、财务、人力等业务域。它给项目的启发是：

- 归档对象可能来自多个业务表和多个文件组件。
- 业务系统中的生命周期状态会影响是否允许归档。
- 归档后仍要能按业务主键回溯来源系统记录。
- 接入边界应明确：哪些数据由源系统负责，哪些数据由档案系统负责。

## 对当前项目的影响

| 项目能力   | 专题启发                                                 |
| ---------- | -------------------------------------------------------- |
| 动态档案表 | 能承载业务字段，但不能替代凭证、附件、明细和审批链路关系 |
| 明细行表   | 适合表达凭证明细、发票明细、业务行等重复结构             |
| 关联档案   | 适合表达报销单、凭证、附件、合同、订单之间的业务关联     |
| 导入接口   | 需要支持来源系统主键、业务单号、幂等和重复校验           |
| 搜索       | 应支持业务关键字段、期间、金额、组织、人员和全文组合查询 |
| 审计       | 应记录来源系统、归档批次、归档人、归档时间和问题处理     |

## 建议进入 OpenSpec 的能力

- 会计档案分类、字段、案卷/件关系和明细行模型。
- 业务系统归档接口的幂等键、来源系统标识和错误返回。
- 报销、凭证、附件、审批记录、入账记录之间的关联模型。
- 归档前校验、归档后查询和证据型导出。
- ERP 接入时源系统与档案系统的数据责任边界。

## 暂不直接采用

- 不把会计专题规则变成所有档案分类的默认规则。
- 不预设所有分类都有案卷/卷内结构；只有业务上确实存在时才建模。
- 不让档案系统直接改写 ERP 或财务系统的业务状态，除非接口合同明确。

## 待确认

- 当前项目是否要优先支持电子会计档案。
- 是否需要接报销、ERP、财务、影像或 OA 系统。
- 会计凭证和凭证明细是作为档案条目、明细行，还是专题模型。
- 是否需要支持电子凭证证据包、凭证版式文件和结构化数据一并导出。
