# 档案专业术语

本页用于统一项目里的档案专业语言。主要参考 `DA/T 1-2000 档案工作基本术语`、`DA/T 58-2014 电子档案管理基本术语`、`DA/T 13-2022 档号编制规则`、`DA/T 18-2022 档案著录规则`、`DA/T 22-2015 归档文件整理规则`。

这里使用项目化释义，不复制标准原文。正式合同应在 OpenSpec 中明确字段、状态和验收。

## 基础对象

| 术语     | 英文建议                                | 项目化理解                                                       | 建模提示                                           |
| -------- | --------------------------------------- | ---------------------------------------------------------------- | -------------------------------------------------- |
| 档案     | archive / archives                      | 在业务活动中形成并具有保存价值的记录集合或记录对象               | 项目领域名词优先用 `Archive`，不要和“归档动作”混用 |
| 文件     | record / document                       | 业务活动中形成的原始记录材料，归档前通常仍处于文件或电子文件阶段 | 不等同于存储文件 blob，也不等同于档案条目          |
| 电子文件 | electronic record                       | 以数字形式形成、传输和处理的文件                                 | 接入业务系统时要保存来源、格式、元数据和处理过程   |
| 电子档案 | electronic archive / electronic records | 经归档纳入档案管理体系的电子文件及相关元数据、组件和过程信息     | 需要比普通附件更强的校验、审计、保存和利用能力     |
| 档案实体 | archival entity                         | 作为管理对象存在的档案实物或电子对象                             | 可映射到案卷、件、文件对象、封装包等               |
| 档案信息 | archival information                    | 档案内容、结构、背景、管理过程和技术属性的组合                   | 不应只落在动态字段表里                             |

## 来源与组织

| 术语     | 英文建议                | 项目化理解                                               | 建模提示                                       |
| -------- | ----------------------- | -------------------------------------------------------- | ---------------------------------------------- |
| 立档单位 | fonds constituting unit | 形成某个全宗的机构、组织或个人                           | 可以作为全宗来源主体，不一定等同于系统组织机构 |
| 全宗     | fonds                   | 一个立档单位形成的、具有有机联系的档案整体               | 当前项目可作为档案数据范围和分类组织的重要维度 |
| 全宗号   | fonds number            | 标识全宗的代码                                           | 档号组成项之一，不应随意用组织 ID 替代         |
| 全宗卷   | fonds descriptive file  | 记录和说明某个全宗历史、管理和现状的专门案卷             | 更偏档案管理工具，不是普通案卷                 |
| 档案门类 | archival category       | 文书、科技、会计、照片、录音、录像、实物、业务数据等门类 | 不等同于项目里的“档案分类”，后者可能更细       |
| 类别号   | series number           | 按分类方案赋予档案类别或层级的代码                       | 档号结构中常见，需要与分类代码边界清楚         |

## 层级与整理

| 术语     | 英文建议             | 项目化理解                                                   | 建模提示                                               |
| -------- | -------------------- | ------------------------------------------------------------ | ------------------------------------------------------ |
| 整理     | arrangement          | 按规则对档案进行分类、组合、排列、编号和编目的过程           | 是管理过程，不是单纯排序                               |
| 分类     | classification       | 按来源、时间、内容、形式、门类或业务规则组织档案             | 项目中的档案分类应明确是业务分类、门类还是技术表路由   |
| 案卷     | file / archival file | 由相互联系的文件材料组成的保管单位                           | 当前可对应 `ArchiveVolume`，但并非所有分类都有案卷层级 |
| 卷内文件 | item within file     | 案卷内部的文件或条目                                         | 当前可对应挂接到案卷的 `ArchiveItem`                   |
| 件       | item                 | 以件为单位整理时的基本保管和著录单位                         | 当前多数无案卷结构的档案条目可按件理解                 |
| 组件     | component            | 组成一个电子文件或电子档案的多个文件、附件、版式件、数据件等 | 适合由文件对象关系表达，不建议塞进单个字段             |
| 附件     | attachment           | 附属于主文件或主档案对象的材料                               | 需要区分业务附件、归档组件、派生预览和证据材料         |
| 关联档案 | related archive      | 业务上相互关联但不属于同一层级的档案对象                     | 应走关系表，不混入父子层级                             |

## 编号与检索

| 术语     | 英文建议               | 项目化理解                                         | 建模提示                                     |
| -------- | ---------------------- | -------------------------------------------------- | -------------------------------------------- |
| 档号     | reference code         | 用于固定和识别档案排列顺序的一组代码               | 不只是展示字段，应有组成规则和唯一性边界     |
| 案卷号   | file number            | 标识案卷顺序或位置的编号                           | 只在案卷结构中自然存在                       |
| 件号     | item number            | 标识件顺序或位置的编号                             | 按件整理时常用，不等同数据库主键             |
| 页号     | page number            | 纸质或版式材料内部页次                             | 电子文件可能更适合记录页数、页范围或组件序号 |
| 目录     | catalog                | 用于检索和管理档案的目录数据集合                   | 可来自著录数据和索引，不等同全文索引         |
| 检索     | retrieval / search     | 按目录、元数据、全文或组合条件查找档案             | 目录检索和全文检索应分层                     |
| 著录     | description            | 为管理和检索而分析、选择、组织并记录档案信息的过程 | 可映射为著录项、元数据项、目录字段和展示字段 |
| 著录项   | description element    | 著录过程中记录的具体信息项                         | 需要和动态字段、系统字段、搜索字段明确映射   |
| 多级著录 | multilevel description | 按全宗、类别、案卷、件等层级分别著录               | 适合有层级的档案，不应强迫无层级分类套用     |

## 生命周期

| 术语 | 英文建议                   | 项目化理解                                   | 建模提示                                  |
| ---- | -------------------------- | -------------------------------------------- | ----------------------------------------- |
| 收集 | acquisition / collection   | 将分散形成的文件材料纳入档案管理准备范围     | 可以对应导入、接入、采集任务              |
| 归档 | filing / archiving process | 把应保存的文件及元数据纳入档案管理体系的过程 | 类名可用 `Filing`，动作方法可用 `fileXxx` |
| 移交 | transfer                   | 形成单位或系统向档案管理方移交档案和相关信息 | 正式能力应建模移交批次、清单、检测和回执  |
| 接收 | accession / receipt        | 档案管理方对移交内容进行校验、登记和入库     | 不等同上传成功                            |
| 鉴定 | appraisal                  | 判断档案价值、保管期限和处置方式的活动       | 后续可影响保管期限、开放、销毁            |
| 保管 | preservation / custody     | 对档案进行保存、维护和安全管理               | 包括对象存储、备份、校验、格式维护和权限  |
| 利用 | access / use               | 查询、阅览、复制、下载、借阅、导出等使用活动 | 高风险利用行为需要权限和审计              |
| 开放 | access opening             | 按规则向更大范围提供利用                     | 和普通授权不同，常受期限、密级和政策影响  |
| 销毁 | destruction                | 对失去保存价值且获准处置的档案进行毁灭性处理 | 需要清册、审批和审计，不是普通删除        |

## 期限、安全与价值

| 术语     | 英文建议                    | 项目化理解                                   | 建模提示                             |
| -------- | --------------------------- | -------------------------------------------- | ------------------------------------ |
| 档案价值 | archival value              | 档案对机构、社会或个人的凭证、参考和历史价值 | 影响是否归档、保管期限和利用控制     |
| 保管期限 | retention period            | 档案应保存的期限，如永久、定期等             | 应作为业务语义字段，不只作为展示文本 |
| 密级     | security classification     | 档案保密等级或敏感程度                       | 与权限、开放和导出控制相关           |
| 权限     | permission / access control | 用户对档案对象和操作的访问控制               | 要区分功能权限、数据范围和状态约束   |
| 审计     | audit                       | 对关键操作、状态变化和利用行为进行留痕       | 不能只依赖应用日志                   |
| 清册     | list / register             | 对档案、移交、销毁、备份等对象形成的正式清单 | 适合建模为批次明细或导出材料         |

## 电子档案专题术语

| 术语     | 英文建议                                        | 项目化理解                                       | 建模提示                               |
| -------- | ----------------------------------------------- | ------------------------------------------------ | -------------------------------------- |
| 元数据   | metadata                                        | 描述档案内容、结构、背景、管理和技术属性的数据   | 分描述、结构、管理、技术、保存等类别   |
| 封装     | package / encapsulation                         | 将文件、元数据、结构关系和校验信息组织成可交换包 | 归档包、移交包、证据包要分别定义       |
| 归档包   | filing package                                  | 用于归档接收的一组文件、元数据和清单             | 不等同普通压缩包                       |
| 移交包   | transfer package                                | 用于正式移交接收的文件、元数据、清单和校验材料   | 应配套接收检测和回执                   |
| 证据包   | evidence package                                | 用于证明档案真实性、完整性和过程可信的导出材料   | 应包含文件、元数据、摘要和必要过程记录 |
| 四性     | authenticity / integrity / usability / security | 真实性、完整性、可用性、安全性的合称             | 可拆成检测项、验收项和审计项           |
| 摘要     | digest / checksum                               | 对文件或包内容计算得到的校验值                   | 必须记录算法名，不只记录字符串         |
| 电子签名 | electronic signature                            | 用于确认签署者身份和内容完整性的电子签署数据     | 是否接入要进入具体业务规格             |
| 时间戳   | timestamp                                       | 证明某一数据在某时刻已经存在或发生的时间凭据     | 证据效力场景可能需要可信时间源         |
| 格式迁移 | format migration                                | 为长期可读把文件转换到新格式的过程               | 需要保留来源、版本、校验和迁移记录     |

## 和当前项目命名的建议

| 中文概念      | 推荐英文              | 不建议                                      |
| ------------- | --------------------- | ------------------------------------------- |
| 档案领域      | `Archive`             | 把 `Archive` 当所有“归档动作”动词           |
| 档案条目 / 件 | `ArchiveItem`         | `ArchiveRecord` 继续承载所有层级            |
| 案卷          | `ArchiveVolume`       | 用泛化 `parent` 表达所有案卷关系            |
| 全宗          | `Fonds`               | 简化成 `Organization`                       |
| 档号          | `ReferenceCode`       | 简化成数据库 `id`                           |
| 著录          | `Description`         | 和 UI 展示说明 `description` 混用时不加限定 |
| 归档流程      | `Filing`              | `ArchiveAction`                             |
| 执行归档      | `fileXxx`             | `filingXxx`、`archiveArchiveXxx`            |
| 已归档状态    | `archived` 或 `filed` | 同一状态在多处混用两个词                    |

## 当前项目需要避免的混淆

- 档案分类不必然等于档案门类；门类是行业分类维度，项目分类可能是业务建模和动态表路由维度。
- 案卷、件、卷内文件是档案整理层级，不是所有分类都必须具备的数据库层级。
- 档号不是数据库主键；数据库主键负责技术定位，档号负责档案管理识别和排列。
- 元数据不是全部动态字段；动态字段只是项目承载部分著录和业务字段的方式。
- 归档、移交、接收、导入不是同一个动作；导入可以是技术入口，归档和移交是业务流程。
- 附件、组件、文件对象、封装包和证据包应按用途区分，不建议统一叫 attachment。

## 待进入 OpenSpec 的术语决策

- `ArchiveItem` 是否明确等价于“件/条目”，以及它和案卷、卷内文件的关系。
- `ArchiveVolume` 是否只表示案卷，不表示所有父节点。
- 全宗、档案分类、档案门类、动态表之间的映射关系。
- 档号组成项、生成规则、唯一性边界和人工调整规则。
- “已归档”使用 `archived` 还是 `filed`，并统一前后端枚举和文案。
- 著录项、动态字段、搜索字段和详情展示字段之间的映射边界。
