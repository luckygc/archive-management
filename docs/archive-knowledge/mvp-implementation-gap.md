# 档案系统 MVP 实施缺口

本文把 [档案系统能力地图](./archive-system-capability-map.md) 中的 MVP 目标落成阶段性缺口清单。它不是接口合同；字段、状态、权限和验收场景以 `openspec/specs/` 及当前 OpenSpec change 为准。

## 判断口径

产品 MVP 的目标是让早期用户持续维护真实档案，而不是只完成演示链路：

- 一个部门可以在后端权限和数据范围约束下维护全宗、分类、档案、案卷、关系、明细和电子文件。
- 大于单页的数据可以稳定浏览，读取失败可以在原工作区恢复，关键写入有明确反馈。
- 批量导入导出、文件预览、归档接收和审批流程必须形成可验收的业务闭环，不能仅以底层接口、独立服务或引擎已存在判定完成。

## 本轮 PC 档案管理闭环

`close-archive-pc-mvp` 已关闭档案日常管理主路径中的以下缺口：

| 能力 | 当前闭环 |
| --- | --- |
| 档案搜索与管理 | 管理查询和全文发现均使用稳定游标分页；草稿条件与已提交查询分离，筛选、排序和页大小变化从第一页开始 |
| 生命周期与字段 | PC 可查看、创建、编辑、锁定、解锁和删除档案；编辑覆盖密级、保管期限、实物字段和动态字段 |
| 案卷 | PC 可筛选、创建和查看案卷，并通过游标搜索查看卷内档案、选择候选档案加入案卷 |
| 关系 | PC 可分页读取关系、从可见档案搜索结果选择目标，并创建或删除关系 |
| 明细 | 分类配置可维护并构建明细表定义；档案编辑可分页维护明细行 |
| 权限与数据范围 | 路由、菜单、页签和按钮复用同一权限快照；服务端继续逐次校验功能权限、数据范围和状态 |
| 工作台 | 摘要来自当前用户数据范围内的真实档案、草稿、锁定和电子文件聚合，不再展示虚构待办 |
| 失败恢复 | 档案搜索、管理、案卷、工作台及资源 Drawer 在原位显示错误；重试保留上下文，游标失效从相同查询第一页恢复 |

这表示“档案 PC 日常管理闭环”完成，不表示整个档案产品 MVP 已全部完成。

## 产品 MVP 仍有缺口

| 缺口 | 当前基础 | 仍需形成的闭环 |
| --- | --- | --- |
| 导入导出 | 已有同步模板、导入和导出接口及基础页面入口 | 面向真实批量数据的任务化执行、进度与失败恢复、逐行校验错误文件回填、权限审计和容量验收 |
| 文件预览 | 已有独立 `backend/preview-service/` 转换服务和电子文件列表、下载能力 | PC 电子文件预览入口、格式能力协商、转换失败反馈、访问授权和部署联调 |
| 归档接收 | `intake` 目前只有模块与概览入口 | 提交、校验、接收、退回、状态流转、材料交接和 PC 操作闭环 |
| 审批流程 | 已接入 Flowable process engine 基础设施 | 档案业务流程定义、任务分派、待办页面、权限、审计和异常恢复；引擎存在不等于审批能力完成 |
| 系统交付配置 | 已有认证、授权、存储和运行时基础设施 | 面向部署人员的必要配置校验、运维文档和目标环境验收；不扩展为通用配置平台 |

## 后续边界

- 不在本轮顺手增加完整档号规则引擎、全宗沿革、长期保存格式迁移、证据包或利用审批。
- 导入导出、预览、接收和审批分别通过独立 OpenSpec change 明确业务状态、异步任务、错误恢复和验收数据，再进入实现。
- 已有接口或基础设施可以复用，但不能用占位入口、同步样例链路或独立服务健康检查替代端到端产品验收。
