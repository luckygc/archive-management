# 法规底座

本页记录档案法律法规对系统设计的底层影响。这里不替代法律意见，也不复制法规全文；项目落地仍以客户场景、合同和 OpenSpec 为准。

## 主要来源

- 中华人民共和国档案法
- 中华人民共和国档案法实施条例
- 机关档案管理规定
- 电子档案管理系统基本功能规定

## 对系统的核心影响

### 档案不是普通附件

档案系统处理的是具有保存价值、凭证价值和利用价值的信息资源。系统设计不能只把文件当作对象存储里的 blob，还要管理：

- 来源、形成者、形成时间、业务活动和责任主体。
- 档案分类、全宗、保管期限、密级或开放控制。
- 原始文件、元数据、过程记录、检测结果和利用记录之间的关系。
- 保管、利用、移交、销毁或长期保存的生命周期。

### 真实、完整、可用、安全是长期约束

项目里的“导入成功”“保存成功”不能等同于档案合规。更稳的验收视角是：

- **真实**：能说明档案来源、形成过程、责任主体和后续变更。
- **完整**：文件、元数据、组件、附件、关联关系和过程记录没有缺失。
- **可用**：能检索、读取、展示、导出，并能在合理范围内解释格式和结构。
- **安全**：权限、审计、备份、恢复、保密和防篡改机制可追踪。

### 信息化能力要服务档案治理

电子档案系统不是单纯的后台 CRUD。法规和规章给项目的启发是：系统能力需要覆盖收集、整理、保管、利用、统计、鉴定、移交和安全，而不是只覆盖录入和查询。

对当前项目的直接影响：

- `archive item`、`archive volume`、文件对象、元数据、关联档案和明细行表要能表达档案业务身份。
- 权限控制不能只按菜单控制，还要能按分类、全宗、状态、密级、数据范围等约束档案访问。
- 审计日志、状态流转和导入导出记录要能支撑追溯。
- 长期保存、备份恢复和格式迁移不能完全交给对象存储实现。

## 项目采纳建议

| 方向         | 建议                                                                 |
| ------------ | -------------------------------------------------------------------- |
| 档案对象建模 | 在 OpenSpec 中明确“档案条目、案卷、文件对象、元数据、关系”的边界     |
| 生命周期     | 按业务场景逐步建模收集、整理、归档、移交、接收、利用、销毁或长期保存 |
| 权限         | 功能权限和数据范围分开，不让前端筛选替代服务端授权                   |
| 审计         | 关键状态变更、导入导出、移交接收、检测、修复和利用都应可追踪         |
| 安全         | 备份、恢复、校验、防篡改和证据效力维护要进入高风险场景验收           |

## 待确认

- 当前产品首期是否面向机关、企业，还是通用档案管理系统。
- 是否要声明支持“单套制”电子档案管理。
- 是否需要按客户行业固化特定归档范围和保管期限表。
- 是否需要对接档案馆移交、政务服务事项归档、ERP 归档或会计档案专题。
