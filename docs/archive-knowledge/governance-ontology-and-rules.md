# 通用本体、本地规则与治理方案

本文用于固化“面向档案馆、政府、企业、高校和业务系统归档”的通用抽象。它是项目知识库和设计基线；已经迁入 OpenSpec 的内容，以对应 `openspec/specs/*` 为实现验收依据。

## 当前项目状态

`add-archive-governance-foundation` 已将三层底座迁入 OpenSpec，并完成第一版可运行闭环：

- `archive-governance-scheme`：治理方案、方案版本、状态流转、适用范围、绑定和默认版本解析。
- `archive-ontology-core`：对象类型、属性类型、属性映射、关系类型和事件类型，并约束映射到固定字段、动态字段、明细字段、文件组件字段和过程字段。
- `archive-local-rule-engine`：规则定义、结构化条件树、字段事实解析、操作符/effect 白名单、规则执行决策和追踪查询。

第一版实现保持原有档案数据面不变：档案实例仍保存到 `am_archive_item`、`am_archive_volume`、分类动态表、明细表和文件组件相关表；本体和规则只解释语义与行为，不保存档案题名、文号、责任者、摘要等实例值。

尚未进入本次实现的能力包括：完整分类方案版本、著录方案、档号流水、保管期限处置、正式归档移交、四性检测任务和行业模板。这些应继续通过独立 OpenSpec change 扩展。

## 核心结论

通用档案平台不应在“通用本体 + 本地规则”和“治理方案聚合”之间二选一。稳定结构应是三层合体：

```text
治理方案 ArchiveGovernanceScheme
    负责客户、机构、行业和场景采用哪套配置

本体 Ontology
    负责统一解释对象、属性、关系、事件和层级

本地规则 LocalRuleEngine
    负责表达本地差异、标准约束、档号、保管、利用和检测行为
```

本体定义语言，规则定义行为，治理方案定义某个机构在某个版本里采用哪套语言和行为。

## 为什么不能只选一种

| 方案 | 优点 | 风险 | 项目判断 |
| ---- | ---- | ---- | -------- |
| 只做通用本体和规则 | 表达力强，能覆盖多行业、多标准和本地差异 | 容易变成万能 EAV、自由脚本或知识图谱式过度设计，查询和治理失控 | 适合作为底座，但必须受控 |
| 只做治理方案聚合 | 交付边界清楚，分类、档号、著录、移交等业务包容易理解 | 如果没有统一本体和规则，会逐渐堆出多个互不兼容的 profile/policy | 适合作为外层聚合，不适合作为唯一抽象 |
| 三层合体 | 既有统一语义，又有本地差异，还能按客户版本交付 | 第一版设计要覆盖全域骨架，建模成本高 | 推荐作为完整平台方向 |

第一版如果定位为完整平台底座，应做“全域可运行最小闭环”，不是只做案卷、卷内和少量校验；但也不应一次穷尽所有行业细则。当前已落地的是治理方案、本体核心和本地规则引擎三层 foundation，下面表格中的分类方案、著录方案、档号、移交和四性等仍作为后续扩展域。

## 标准和实践输入

当前抽象至少要能承接以下类型的规范影响：

- `DA/T 18-2022 档案著录规则`：要求系统支持多级著录、著录项、层级性、关联性和动态维护。
- `DA/T 13-2022 档号编制规则`：要求档号由全宗号、类别号、案卷号、件号、页号等组成项按规则生成和校验。
- `DA/T 22-2015 归档文件整理规则`：要求按件整理、分类、排列、编号、编目和电子文件元数据收集。
- `DA/T 93-2022 电子档案移交接收操作规程`：要求移交包、目录数据、内容数据、元数据、检测、签收、登记和入库。
- 电子档案和长期保存相关规范：要求区分描述元数据、结构元数据、管理元数据、技术元数据、权限利用元数据和保存元数据。

这些规范共同说明：档案系统不是“档案记录表 + 附件表”，也不是“所有字段都动态配置”。更稳定的模型要同时表达对象、层级、元数据、文件组件、过程、规则、检索和审计。

## 第一版全域最小闭环

如果第一版按完整平台底座设计，应覆盖这些能力的最小闭环：

| 能力 | 第一版应有的闭环 |
| ---- | ---------------- |
| 治理方案 | 方案、版本、适用范围、发布、冻结、复制和退役 |
| 本体核心 | 对象类型、属性类型、关系类型、事件类型、字典和版本 |
| 分类方案 | 分类树、分类版本、分类到对象类型、字段、规则和动态表的绑定 |
| 著录方案 | 著录层级、著录项、必著/选著、适用主体和目录展示 |
| 档号规则 | 组成项、流水作用域、生成、校验、占号、冲突处理和历史 |
| 保管期限 | 归档范围、期限、到期提醒、处置动作和开放鉴定触发 |
| 文件组件 | 主文件、附件、版式件、派生件、摘要、格式、组件顺序和来源 |
| 归档与移交 | 归档批次、移交包、清单、检测、签收、登记和入库 |
| 四性检测 | 检测项、检测任务、结果、问题、修复建议和证据记录 |
| 利用控制 | 功能权限、数据范围、密级、开放状态、利用登记和审计 |
| 本地规则 | 校验、派生、档号、保管、访问、检测、移交、归档和导出规则 |

这里的“全部实现”指每个能力域都有稳定抽象、数据落点、规则入口和可验收最小行为，不表示第一版内置所有行业模板和所有地方细则。

## 档案层级

档案领域存在多种层级，但不是每个层级都应成为档案主对象。

```text
立档单位 / 形成主体
└── 全宗
    └── 分类方案 / 分类节点
        ├── 门类、子门类、业务分类
        ├── 年度、保管期限、机构/问题、载体等分类维度
        └── 可选业务聚合层
            ├── 项目、案件、工程、课题、活动、专题
            └── 案卷 ArchiveVolume，可选
                └── 件 / 卷内 ArchiveItem
                    ├── 明细行 Line rows
                    └── 文件组件 File components
                        ├── 正文、附件、版式件、OCR、预览、派生件
                        └── 页、张、帧、时间片段等内部结构
```

### 层级身份判断

| 概念 | 是否档案主对象 | 建模方式 |
| ---- | -------------- | -------- |
| 立档单位 | 否 | 来源主体或组织主体 |
| 全宗 | 半主对象 | 全宗治理、数据范围、档号组成项和来源追溯 |
| 门类、子门类、业务分类 | 否 | 分类方案树 |
| 年度、保管期限、机构/问题、载体 | 否 | 分类维度、规则字段或档号组成项 |
| 项目、案件、工程、课题、活动、专题 | 看场景 | 业务聚合对象或关系对象，不默认升为档案层级 |
| 案卷 | 是 | `ArchiveVolume` |
| 件 / 卷内文件 | 是 | `ArchiveItem` |
| 明细行 | 通常否 | line table；只有单行有独立生命周期、权限和检索入口时才考虑升格 |
| 文件组件 | 否 | 文件组件表，挂到 item 或 volume |
| 页、张、帧、时间片段 | 否 | 组件内部结构或技术元数据 |
| 归档包、移交包、证据包 | 否 | 过程对象或交换对象 |
| 目录、清册 | 否 | 检索、管理输出或批次明细 |
| 关联档案 | 否 | 关系表，不混入父子层级 |

项目实现层应保持 `ArchiveVolume` 可选、`ArchiveItem` 必有。更多领域层级进入本体和关系模型，不应把主表重新泛化为单一 `ArchiveRecord`。

## 存储模型

通用本体不等于把档案存成三元组表或万能 JSON。推荐把存储拆成控制面、数据面和检索面。

```text
控制面：解释数据是什么
├── governance_scheme
├── ontology_concept_type
├── ontology_attribute_type
├── ontology_relation_type
├── description_profile
├── reference_code_rule_set
└── rule_definition

数据面：保存真实档案事实
├── am_archive_volume
├── am_archive_item
├── 分类动态表
├── 明细行表
├── 文件组件表
├── 关系表
└── 过程、检测、移交、利用、审计表

检索面：面向查询和发现
├── 管理查询：主表 + 分类动态表 + 结构化条件
├── 全文发现：item_id + search_text + index_version
└── 统计报表：按需构建汇总或投影
```

### 字段落库原则

| 字段类型 | 落点 |
| -------- | ---- |
| 高频固定字段 | 固定主表列，例如全宗号、分类、档号、状态、密级、保管期限 |
| 分类业务字段 | 分类动态表 typed columns |
| 可重复业务明细 | 明细行表 typed columns |
| 文件技术属性 | 文件组件或文件对象表 |
| 规则定义、条件树、表达式 | 受控 `jsonb` 配置，不直接作为查询 SQL |
| 低频扩展配置 | 受控 `jsonb`，但不能绕过字段注册和类型校验 |
| 全文内容 | 全文投影表或全文 provider 索引 |

不建议使用以下形态保存主数据：

```text
object_id | attribute_id | value_text | value_number | value_date
```

这种 EAV 形态短期通用，长期会让排序、唯一约束、组合筛选、权限过滤、统计报表和执行计划都变复杂。它可以用于低频补充属性或迁移过渡，不应成为档案主数据形态。

## 检索模型

检索要分成后台管理查询和普通用户发现型搜索。

### 后台管理查询

后台管理查询使用数据库语义：

```text
主表固定字段
    + 分类动态表 typed columns
    + 结构化条件树
    + 数据范围
    + 逻辑删除
    + 排序和游标分页
```

字段能否被精确筛选应由字段定义决定。后端负责把字段编码解析成受控列名，把操作符解析成白名单 SQL 片段，并使用参数绑定传值。客户端不能提交自由 SQL、SpEL、MVEL 或 JavaScript。

### 全文发现

普通用户全文发现使用独立投影：

```text
ArchiveItem
    + 动态字段文本
    + 明细行文本
    + 文件 OCR 或正文抽取
    + 权限和状态过滤
    -> 全文投影
```

全文投影锚定 `ArchiveItem`。明细行文本可以进入 item 搜索文档，但明细行本身不默认升为顶层档案记录。

### 跨分类查询

跨分类查询分三类处理：

| 查询类型 | 处理方式 |
| -------- | -------- |
| 全宗、分类、状态、密级、保管期限等固定字段 | 直接查主表 |
| 题名、形成时间、责任者等本体共通属性 | 通过本体属性映射到各分类字段后查询或走专门投影 |
| 某个分类特有字段 | 必须限制分类范围后查询对应动态表 |

不要承诺“任意动态字段跨所有分类自由筛选”。跨分类精确筛选必须有本体属性映射、字段类型一致性和索引策略支撑。

## 本体模型

本体只解释语义，不替代物理存储。

### ConceptType

对象类型定义系统能识别哪些领域对象：

| 类型 | 说明 |
| ---- | ---- |
| `FONDS` | 全宗 |
| `CLASSIFICATION_SCHEME` | 分类方案 |
| `CLASSIFICATION_NODE` | 分类节点 |
| `DESCRIPTION_UNIT` | 著录单元 |
| `ARCHIVE_VOLUME` | 案卷 |
| `ARCHIVE_ITEM` | 件或卷内文件 |
| `ARCHIVE_LINE` | 明细行 |
| `FILE_COMPONENT` | 文件组件 |
| `TRANSFER_PACKAGE` | 移交包 |
| `FILING_BATCH` | 归档批次 |
| `QUALITY_CHECK` | 四性检测任务或检测项 |
| `ACCESS_EVENT` | 利用事件 |
| `CATALOG` | 目录或清册 |

### AttributeType

属性类型定义字段语义：

| 属性内容 | 说明 |
| -------- | ---- |
| 编码和名称 | 稳定业务编码和显示名称 |
| 数据类型 | 文本、数字、日期、时间、枚举、布尔、金额、组织、人员、引用等 |
| 适用对象 | volume、item、line、component、package 或 process |
| 字段域 | 描述、结构、管理、技术、权限利用、保存 |
| 基数 | 单值、多值、可重复行 |
| 是否允许精确筛选 | 影响索引和管理查询 |
| 是否参与著录 | 影响目录和著录方案 |
| 是否参与档号 | 影响档号规则 |
| 是否参与规则事实 | 影响本地规则引擎可见范围 |

全文投影不作为字段级开关暴露给用户。是否进入全文应由投影策略和字段域决定，避免让每个字段配置出不可维护的搜索行为。

### RelationType

关系类型定义对象之间如何关联：

| 类型 | 说明 |
| ---- | ---- |
| `BELONGS_TO` | 隶属，例如 item 属于 fonds 或 category |
| `CONTAINS` | 包含，例如 volume 包含 item |
| `HAS_COMPONENT` | 拥有文件组件 |
| `HAS_LINE` | 拥有明细行 |
| `RELATED_TO` | 平级关联档案 |
| `DERIVED_FROM` | 派生来源 |
| `TRANSFERRED_IN` | 来自某次移交 |
| `GENERATED_BY` | 由某过程生成 |
| `DESCRIBED_BY` | 被某著录方案描述 |

父子层级只能表达真实从属关系。业务关联必须走关系表，不能混进 `parent_id`。

### EventType

事件类型用于解释过程和审计：

| 类型 | 说明 |
| ---- | ---- |
| `CREATED` | 创建 |
| `UPDATED` | 修改 |
| `FILED` | 归档 |
| `TRANSFERRED` | 移交 |
| `ACCESSED` | 查阅 |
| `DOWNLOADED` | 下载 |
| `EXPORTED` | 导出 |
| `CHECKED` | 检测 |
| `APPRAISED` | 鉴定 |
| `DISPOSED` | 处置 |

事件不只用于日志，也应能被规则、审计和证据包引用。

## 本地规则引擎

本地规则引擎是受控解释器，不是自由脚本引擎。

### 规则结构

规则至少包含：

| 字段 | 说明 |
| ---- | ---- |
| `id`、`code`、`name` | 规则身份 |
| `ruleType` | 规则类型 |
| `trigger` | 触发点 |
| `scope` | 适用全宗、分类、层级、对象和治理方案版本 |
| `priority` | 同类规则排序 |
| `condition` | 结构化条件树 |
| `effects` | 受控效果列表 |
| `message` | 用户可见说明 |
| `effectiveFrom`、`effectiveTo` | 生效时间 |
| `version`、`enabled` | 版本和启停 |

### 规则类型

第一版应覆盖这些类型：

| 类型 | 用途 |
| ---- | ---- |
| `VALIDATION` | 必填、格式、跨字段一致性、唯一前置校验 |
| `DERIVATION` | 默认值、派生字段、建议值 |
| `REFERENCE_CODE` | 档号组成、流水、占号、校验和冲突提示 |
| `RETENTION` | 保管期限、到期提醒、开放鉴定和处置建议 |
| `ACCESS` | 密级、开放状态、利用范围和导出限制 |
| `QUALITY` | 四性检测项、检测结果和问题等级 |
| `TRANSFER` | 移交包、清单、摘要、元数据完整性检查 |
| `FILING` | 归档前预检、整理、提交、入库条件 |
| `EXPORT` | 导出范围、脱敏、证据包内容和审计要求 |

### 条件树

条件树应是结构化 AST：

```json
{
  "all": [
    {
      "field": "item.documentDate",
      "operator": "GTE",
      "value": "2026-01-01"
    },
    {
      "field": "item.securityLevel",
      "operator": "IN",
      "value": ["INTERNAL", "SECRET"]
    }
  ]
}
```

字段必须解析为以下类型之一：

- 固定字段。
- 分类动态字段。
- 明细行字段。
- 文件组件字段。
- 关系字段。
- 上下文字段，例如当前用户、当前组织、当前时间、当前操作。

操作符必须按字段类型白名单限制。文本、数字、日期、枚举、布尔、引用类型不能共享同一套操作符。

### Effect

规则引擎只返回决策，不直接写数据库、不直接发事件、不直接启动流程。

| Effect | 说明 |
| ------ | ---- |
| `VALIDATION_ERROR` | 阻断并返回错误 |
| `WARNING` | 返回警告，由调用场景决定是否允许继续 |
| `SUGGEST_VALUE` | 建议填充值 |
| `DERIVED_VALUE` | 返回派生值，由用例服务决定是否写入 |
| `REQUIRE_REVIEW` | 要求进入审核或人工确认 |
| `REQUIRE_QUALITY_CHECK` | 要求补充检测 |
| `DENY_ACCESS` | 拒绝利用、下载、导出等动作 |
| `MASK_FIELD` | 要求展示或导出时脱敏 |
| `INCLUDE_IN_PACKAGE` | 要求导出包或移交包包含某类材料 |

调用方必须根据当前用例处理 effect。规则引擎不拥有事务边界。

### 决策追踪

规则执行结果应保留可解释信息：

- 命中的规则。
- 参与判断的事实字段。
- 条件树每个节点的结果。
- 产生的 effect。
- 跳过规则的原因。
- 使用的治理方案版本。

这对档号冲突、移交检测、利用拒绝、开放鉴定和四性检测问题定位都很关键。

## 治理方案

治理方案是客户或机构实际采用的一组本体、字段、规则和流程配置。

```text
ArchiveGovernanceScheme
├── 基本信息：编码、名称、机构、适用场景
├── 本体配置：启用哪些概念、属性、关系和事件
├── 分类方案：分类树、版本、适用范围
├── 著录方案：层级、著录项、目录展示
├── 档号规则：组成项、流水、唯一性、历史
├── 保管期限：归档范围、期限和处置动作
├── 文件组件策略：组件类型、格式、摘要、派生件
├── 归档和移交策略：批次、包、清单、检测和回执
├── 利用控制策略：密级、开放、数据范围、审批和审计
└── 本地规则集：各类 rule packs
```

### 生命周期

治理方案应至少有这些状态：

| 状态 | 说明 |
| ---- | ---- |
| `DRAFT` | 草稿，可编辑，可验证 |
| `PUBLISHED` | 已发布，可用于新档案 |
| `FROZEN` | 已冻结，用于解释历史档案，不再接收新档案 |
| `RETIRED` | 已退役，只保留审计和追溯 |

已归档档案应记录采用的治理方案版本。分类代码、档号规则、著录项含义发生制度性变更时，历史记录不能被新版本重算破坏。

## 与当前项目的关系

当前项目已有一些正确地基：

- `ArchiveVolume` 和 `ArchiveItem` 已拆分，案卷不是 item 的一行。
- `volume_id` 表达 item 到 volume 的挂接，关联档案走关系表。
- `field_scope` 区分描述元数据和实物信息字段。
- 动态表按分类、层级和字段域生成，适合承载 typed columns。
- 精确筛选和全文投影已经分层，全文投影锚定 item。
- 数据范围条件使用受控结构，不允许自由 SQL。

仍需补齐的能力：

- 治理方案和版本。
- 通用本体的概念、属性、关系和事件定义。
- 分类方案版本和分类到动态表、规则、档号的绑定。
- 著录方案和多级著录项。
- 档号规则引擎和档号历史。
- 保管期限、归档范围和处置规则。
- 文件组件策略和长期保存派生件关系。
- 正式归档、移交接收和四性检测模型。
- 本地规则引擎和规则决策追踪。

## OpenSpec 迁移建议

本文定稿后，不建议一次创建一个巨大实现变更。更稳的做法是先创建一个总设计 change，再拆可验收能力。

推荐顺序：

1. `add-archive-governance-foundation`
   - 治理方案、版本、本体核心、规则定义的最小合同。
2. `add-archive-classification-schemes`
   - 分类方案、节点版本、分类到字段、表和规则的绑定。
3. `add-archive-description-profiles`
   - 著录层级、著录项、目录展示和字段映射。
4. `add-archive-reference-code-rules`
   - 档号组成项、流水、占号、校验和历史。
5. `add-archive-local-rule-engine`
   - 条件树、事实解析、操作符白名单、effect 和决策追踪。
6. `add-archive-filing-transfer-quality`
   - 归档、移交、四性检测和问题反馈。

如果要第一版完整交付，也可以把 1 到 5 合并为一个设计 change，但实现任务仍应按模块拆分，避免一次改动跨越过多表、接口和页面。

## 设计守则

1. 本体解释语义，不替代物理存储。
2. 规则表达行为，不拥有事务边界。
3. 治理方案聚合配置，不绕过 OpenSpec 合同。
4. 档案主对象保持 `volume` 和 `item` 清晰，不回退到单一抽象记录。
5. 明细行、文件组件、归档包、移交包、目录和清册不要默认升为档案主对象。
6. 查询走主表、动态表、索引和投影，不直接扫本体配置。
7. 跨分类精确筛选必须有本体属性映射和字段类型一致性。
8. 受控 `jsonb` 只能保存规则、条件和低频配置，不能成为自由 SQL 或万能字段。
9. 历史档案记录必须能解释当时采用的分类、档号、著录和规则版本。
10. 第一版做全域骨架和最小闭环，不把所有地方规范和行业模板硬编码进系统。
