# 档号与全宗规则

本页整理全宗和档号的项目化理解。正式规则必须在具体业务规格中确认，不能把示例当成通用强制格式。

## 全宗规则

全宗的核心是来源原则：同一立档单位形成的、具有有机联系的档案整体，通常作为一个全宗管理。

| 规则                 | 项目理解                                                                               |
| -------------------- | -------------------------------------------------------------------------------------- |
| 全宗不是组织表       | 全宗可能对应历史机构、合并前机构、撤销机构、个人或特殊来源，不应直接用当前组织 ID 代替 |
| 全宗号是档案管理代码 | 全宗号参与档号和检索，和数据库主键分工不同                                             |
| 一个机构可有历史演变 | 学校更名、合并、拆分时，可能需要多个全宗或全宗沿革                                     |
| 一个馆可管理多个全宗 | 高校档案馆常同时管理本校、前身学校、合并单位和专题征集全宗                             |
| 全宗可有层级或群组   | 联合全宗、汇集全宗、全宗群需要表达关系，但不能破坏每个来源的边界                       |
| 全宗影响权限         | 不同全宗可能有不同开放、利用、密级和数据范围规则                                       |

## 档号的作用

档号是档案管理识别和排列代码，不是数据库主键。它至少承担：

- 固定档案在分类、年度、案卷、件或载体序列中的位置。
- 支持排架、检索、引用、移交和利用。
- 表达全宗、门类、年度、保管期限、组织/问题、案卷号、件号等规则项。
- 在同一规则边界内保持唯一，避免重号。

## 常见档号组成项

| 组成项    | 示例             | 是否必需         | 说明                                   |
| --------- | ---------------- | ---------------- | -------------------------------------- |
| 全宗号    | `2`、`A182`      | 多全宗场景常用   | 表示来源整体                           |
| 年度      | `2024`           | 常用             | 可按形成年度、摄影年度、归档年度等区分 |
| 门类代码  | `WS`、`JX`、`ZP` | 常用             | 表示档案门类或分类                     |
| 分类号    | `DQ11`、`JX14`   | 高校和专题常用   | 可由一级、二级、三级类目组成           |
| 保管期限  | `Y`、`D30`、`10` | 文书常用         | 具体代码需由客户规则确认               |
| 机构/问题 | `XZ`、部门编码   | 机关文书常用     | 可能表示形成部门或问题主题             |
| 案卷号    | `23`             | 案卷结构中常用   | 按分类号下的案卷顺序编号               |
| 件号      | `0001`           | 按件整理常用     | 文件级或件级识别                       |
| 组号      | `1`              | 照片、音视频常用 | 一次活动或一组相关材料                 |
| 流水号    | `0001`           | 常用             | 组内、类内或期限内顺序编号             |

## 常见模板

### 高校案卷级模板

《高等学校档案实体分类法》公开页面中的基本模式是：

```text
年度号-分类号-案卷号
```

多全宗场景可在年度前加全宗号；标识到文件级时可在案卷号后加件号。

```text
全宗号-年度号-分类号-案卷号-件号
```

这个模板适合高校传统案卷管理，不代表所有电子档案都应使用案卷号。

### 机关按件整理模板

机关文书和多门类样例常见结构是：

```text
全宗号-档案门类代码·年度-保管期限代码-机构或问题-件号
```

照片、录音、录像、实物等门类可能减少机构/问题项，改用形成时间、组号或载体序号。

### 音像档案模板

清华大学公开的音像档案办法示例中，照片、录像、录音使用类似结构：

```text
全宗号-摄录年度-门类-单位编码-组号-流水号
```

这种规则适合按活动成组管理的音像档案。项目中如果支持照片/视频专题，应保留“组”和“单件”的两级关系。

## 编号生成原则

1. 档号规则必须绑定到分类方案、全宗或业务场景。
2. 每个组成项要有来源：手工录入、字典、组织编码、年度计算、流水生成或外部系统传入。
3. 流水号必须明确作用域，例如同一全宗+年度+门类+保管期限下唯一。
4. 档号生成后原则上冻结；需要变更时应记录原因、旧档号和新档号。
5. 手工调整需要权限和审计，不能让普通编辑绕过唯一性。
6. 档号展示字符串和档号组成项要同时保存，方便检索、排序和重算检查。
7. 档号分隔符、大小写、补零宽度、中文代码和英文代码要由规则配置，不散落在代码里。

## 推荐数据概念

| 概念       | 说明                                               |
| ---------- | -------------------------------------------------- |
| 全宗       | 来源整体，含全宗号、名称、起止时间、历史沿革和状态 |
| 全宗关系   | 前身、合并、拆分、联合、汇集、全宗群等关系         |
| 档号规则   | 由多个组成项构成的模板和唯一性作用域               |
| 档号组成项 | 全宗号、年度、门类、保管期限、机构、案卷号、件号等 |
| 编号流水   | 某个作用域下的顺序号生成器                         |
| 档号历史   | 档号变更前后的追溯记录                             |

## 对当前项目的实现建议

- `reference_code` 保存最终档号字符串。
- `fonds_id` / `fonds_code` 与 `category_id` / `category_code` 分开保存。
- 档号组成项建议结构化保存，避免后续只能解析字符串。
- 对动态表，不要让每张表自定义一套完全不透明的档号逻辑；应通过规则模板统一生成。
- 档号唯一约束应按业务作用域设计，逻辑删除表仍应只约束未删除记录。
- 外部系统传入档号时，系统仍应校验唯一性、格式和全宗/分类匹配。

## 待进入 OpenSpec 的决策

- 当前系统是否内置档号生成，还是只保存外部档号。
- 档号生成是否允许按分类节点配置不同模板。
- 全宗是否作为所有档案必填字段。
- 档号唯一性作用域是全局、全宗内、分类内，还是规则自定义。
- 历史档号变更是否允许，允许时如何审计。
