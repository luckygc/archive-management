# 四性、证据效力与备份

本页关注电子档案的真实性、完整性、可用性、安全性，以及证据效力维护和备份恢复，主要参考 `DA/T 70-2018`、`DA/T 97-2023`、`DA/T 99-2024`、`GB/T 18894-2016` 等。

## 四性理解

项目里可以把四性转换成可实现、可测试的能力。

| 维度   | 系统问题                                           | 可能的项目能力                                   |
| ------ | -------------------------------------------------- | ------------------------------------------------ |
| 真实性 | 这个档案是否来自可信来源，形成和归档过程是否可追溯 | 来源系统、责任人、时间、签名、审计、状态流转     |
| 完整性 | 文件、元数据、组件、附件和关系是否缺失或被破坏     | 必填校验、数量校验、摘要校验、关联校验、问题清单 |
| 可用性 | 多年后是否还能检索、读取、展示和导出               | 格式管理、预览、全文索引、导出包、恢复演练       |
| 安全性 | 是否能防止未授权访问、修改、删除和泄露             | 权限、数据范围、审计、备份、保密、最小权限       |

## 检测不等于单次校验

四性检测不应该只发生在导入瞬间。高风险场景至少可能包括：

- 归档前检测。
- 接收移交包时检测。
- 入库后定期抽检或全量校验。
- 备份恢复后校验。
- 格式迁移或修复后校验。
- 导出证据包前校验。

## 证据效力维护理解

证据效力维护关注的是档案作为证据时能否说明其可信链路。项目上应重点关注：

- 原始文件和元数据的摘要。
- 归档、修改、检测、迁移、利用、导出的审计记录。
- 电子签名、时间戳或外部可信服务的接入边界。
- 导出时能否带出元数据、摘要、日志和说明。
- 是否能证明系统内记录没有被未授权修改。

这不意味着首期必须接入电子签章或时间戳服务，但数据模型应避免把证据链路做死。

## 备份恢复理解

档案数字资源备份不只是数据库备份。当前项目至少要区分：

- 业务数据库。
- 对象存储中的原始文件、附件和派生对象。
- 搜索索引和可重建投影。
- 配置、字典、字段定义、权限和流程定义。
- 审计日志、归档批次、检测结果和任务结果。

恢复验收不能只看服务能启动，还要抽样验证档案、文件对象、元数据、索引和权限是否一致。

## 对当前项目的影响

| 能力     | 建议                                                     |
| -------- | -------------------------------------------------------- |
| 文件摘要 | 对归档文件、移交包和证据包使用明确摘要算法，并保存算法名 |
| 检测结果 | 用结构化结果保存检测项、等级、问题位置和修复建议         |
| 审计     | 关键业务操作用业务审计表或事件记录，不只依赖应用日志     |
| 备份     | 数据库和对象存储要成对考虑，搜索索引可标记为可重建       |
| 导出     | 证据型导出应包含文件、元数据、摘要和必要过程记录         |
| 权限     | 下载、导出、删除、修复和迁移比普通查看风险更高           |

## 建议进入 OpenSpec 的能力

- 四性检测任务和检测结果 DTO。
- 归档、移交、接收、备份恢复后的校验规则。
- 证据包导出格式和内容范围。
- 备份恢复验收清单。
- 高风险操作审计字段和保留策略。

## 暂不直接采用

- 不在没有真实需求前自研完整电子签章、时间戳或区块链存证能力。
- 不把所有检测项都做成强制阻断；应按风险区分错误、警告和提示。
- 不让前端提示替代服务端校验、摘要校验和审计记录。
