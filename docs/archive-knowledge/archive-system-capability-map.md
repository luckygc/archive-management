# 档案系统能力地图

本文用于多轮对话时对齐“档案系统到底要做什么”。它不是 OpenSpec 合同，也不是开发执行计划；它是领域能力地图和路线图基线。

## 文档落点

| 文档类型                   | 适用内容                                             | 是否适合本文               |
| -------------------------- | ---------------------------------------------------- | -------------------------- |
| `docs/archive-knowledge/*` | 行业知识、能力地图、术语、路线图、候选理解           | 适合                       |
| `openspec/specs/*`         | 已决定要实现的业务合同、接口行为、状态机、验收场景   | 不适合直接写完整愿景       |
| `openspec/changes/*`       | 某次明确变更的提案、设计、任务和规格增量             | 等确定某一阶段要做时再创建 |
| `docs/superpowers/plans/*` | 已确认需求后的执行计划，包含具体文件、步骤、验证命令 | 不适合做长期能力地图       |
| `AGENTS.md`                | 协作规则、架构边界、工具链、代码风格                 | 不适合写业务能力清单       |

结论：本文放在 `docs/archive-knowledge/`。后续如果决定实现某一层能力，再拆成 OpenSpec change；实现计划再写 superpowers plan。

## 分层定义

| 层级     | 目标                                     | 典型用途                         |
| -------- | ---------------------------------------- | -------------------------------- |
| 最小 POC | 证明模型和关键链路跑得通                 | 领域验证、技术验证、演示         |
| MVP      | 可被早期用户持续使用的最小档案管理产品   | 内部试用、种子客户、基础交付     |
| 基础能力 | 一个档案系统应具备的稳定地基             | 正式项目、可维护交付、后续扩展   |
| 完整能力 | 面向档案馆、行业客户和长期运行的完整平台 | 大客户、长期保存、监管和复杂业务 |

## 能力总览

| 能力域         | 最小 POC           | MVP                       | 基础能力                                 | 完整能力                                 |
| -------------- | ------------------ | ------------------------- | ---------------------------------------- | ---------------------------------------- |
| 机构与全宗     | 单个全宗码表       | 多全宗基础维护            | 全宗沿革、立档单位、全宗关系             | 全宗群、全宗卷、跨机构移交和开放策略     |
| 分类与字段     | 少量手工分类和字段 | 分类树、动态字段、布局    | 分类方案、版本、模板、分类到表/规则映射  | 多行业模板、分类迁移、历史冻结和批量治理 |
| 档号           | 手工录入档号       | 基础唯一校验              | 档号规则、流水、组成项、历史             | 多规则并存、迁移重排、冲突修复和档号审计 |
| 档案对象       | 条目最小 CRUD      | item/volume、关联、明细   | 案卷/件/组件/目录分层稳定                | 多载体、多层级、多专题对象模型           |
| 文件对象       | 上传并关联文件     | 文件元数据、SHA-256、下载 | 主文件、附件、组件、版本、格式           | 格式迁移、派生件、长期保存格式治理       |
| 著录与目录     | 动态字段展示       | 列表、详情、搜索字段      | 著录项、案卷目录、卷内目录、专题目录     | 多级著录、编研、finding aid、开放目录    |
| 归档流程       | 创建即入库         | 简单提交/入库状态         | 收集、预检、整理、著录、审核、入库、退回 | 多来源归档、批次、质量控制、跨系统协同   |
| 接收与移交     | 无或入口声明       | 基础导入                  | 移交批次、清单、检测、问题反馈、回执     | 标准移交包、馆际/系统间移交、重传和追溯  |
| 检索           | 基础列表查询       | 结构化筛选和全文发现      | 目录检索、全文、权限、逻辑删除统一       | 跨库检索、专题检索、语义检索、检索审计   |
| 权限           | 登录后可用         | 菜单和基础数据范围        | 功能权限、全宗/分类/状态/密级数据范围    | 开放控制、利用审批、个人信息和版权限制   |
| 利用服务       | 查看详情           | 下载、导出                | 查档、阅览、借阅、复制、利用登记         | 在线阅览室、利用审批、开放目录、统计分析 |
| 四性检测       | 摘要校验           | 入库前必填和摘要校验      | 真实性、完整性、可用性、安全性检测模型   | 定期检测、修复、证据效力维护和检测报告   |
| 审计与证据     | 创建更新时间       | 基础操作日志              | 业务审计、状态流转、下载导出审计         | 证据包、时间戳/签名接入、不可抵赖链路    |
| 保管期限与处置 | 手工字段           | 保管期限字段              | 归档范围、保管期限表、到期提醒、鉴定     | 销毁清册、审批、开放鉴定、自动处置策略   |
| 备份与长期保存 | 依赖部署备份       | 文件和数据库可备份        | 数据库+对象存储一致备份、恢复验收        | 多副本、离线备份、灾备演练、格式迁移     |
| 统计与运维     | 简单数量           | 分类/全宗数量             | 归档、利用、移交、检测、存储统计         | 监管报表、质量分析、容量预测和 SLA       |

## 最小 POC

最小 POC 的目标是证明当前模型能支撑“建分类、建字段、建表、录入档案、上传文件、搜索出来”这条主链路。

### 必做

- 单个或少量全宗维护。
- 档案分类树维护。
- 动态字段维护和动态表构建。
- 档案条目 `ArchiveItem` 的创建、编辑、删除、详情、列表。
- 可选案卷 `ArchiveVolume`，只验证 item 挂接 volume 的模型。
- 文件上传并关联到档案条目。
- 基础结构化筛选和全文投影。
- 逻辑删除和未删除唯一性。

### 不做

- 档号自动生成。
- 分类方案版本。
- 正式归档审批。
- 移交接收包。
- 四性检测模型。
- 借阅、利用审批、销毁鉴定。
- 长期保存和备份恢复验收。

### 验收口径

- 能配置一个分类并生成动态表。
- 能创建一条档案，保存动态字段和文件。
- 能在列表和全文发现中查到该档案。
- 删除后不再出现在正常查询中，唯一值可释放。

## MVP

MVP 的目标是让早期用户能持续维护一批真实档案，而不是只看演示。

### 必做

- 多全宗基础维护，至少支持全宗编码、名称、启停、排序。
- 分类树、动态字段、字段布局、动态表构建。
- `ArchiveItem` / `ArchiveVolume` 分对象管理。
- 档案关联和明细行表，用于会计凭证明细、项目节点等重复结构。
- 档号字段保存和唯一校验，先允许手工录入。
- 文件对象本地记录、对象存储路由和 SHA-256 指纹。
- 管理查询、精确筛选、排序、全文发现。
- 基础权限：登录、菜单、接口权限、全宗/分类数据范围的预留或最小实现。
- 基础导入导出：至少支持标准 Excel 导入导出或批量创建。
- 基础审计：创建、修改、删除、锁定、下载或导出的关键记录。

### 不做

- 完整档号规则引擎。
- 全宗沿革和全宗关系。
- 分类方案版本迁移。
- 完整归档流程和移交接收。
- 完整四性检测。
- 销毁鉴定和长期保存格式迁移。

### 验收口径

- 一个真实部门能用它维护分类、字段、档案、文件和查询。
- 一批真实档案能导入、校验、查询、导出。
- 权限不靠前端隐藏按钮兜底。
- 文件下载不依赖全局存储配置猜历史位置。

## 基础能力

基础能力是正式档案系统的地基。它不追求大而全，但必须减少后续返工。

### 必做

- 分类方案：方案、节点、版本、适用范围、启停和历史冻结。
- 全宗治理：立档单位、全宗号、全宗名称、起止时间、沿革说明和关系。
- 档号规则：模板、组成项、流水作用域、补零、分隔符、唯一性、手工调整和历史。
- 保管期限表：归档范围、保管期限、处置动作、到期提醒。
- 正式归档流程：收集、预检、整理、著录、提交、审核、入库、退回、问题清单。
- 电子文件组件：主文件、附件、版式件、派生件、版本、格式、摘要和组件顺序。
- 著录和目录：著录项模板、案卷目录、卷内目录、文件级目录。
- 四性检测：检测任务、检测项、结果、等级、问题定位和修复记录。
- 移交接收：移交批次、清单、接收检测、问题反馈、回执。
- 利用服务：查档、阅览、下载、复制、导出、利用登记。
- 业务审计：归档、修改、删除、检测、下载、导出、移交、接收。
- 备份恢复验收：数据库和对象存储一致性、恢复抽检和摘要校验。

### 不做

- 不必一开始做复杂馆际交换。
- 不必一开始做电子签章、时间戳或区块链存证。
- 不必一开始覆盖所有行业专题模板。
- 不必一开始做高级编研、展陈、开放门户。

### 验收口径

- 档案分类、全宗、档号、保管期限、归档流程不互相漂移。
- 正式档案能追溯来源、归档过程、文件组件、元数据、利用和审计。
- 可以解释一条档案为什么存在、属于哪个全宗/分类、档号如何生成、保存多久、谁看过、能否开放。

## 完整能力

完整能力面向长期运行的档案馆或大型行业客户。

### 能力范围

- 多机构、多全宗、多库房、多系统接入。
- 标准化移交包、归档包、证据包、开放包。
- 电子签名、可信时间戳、证据效力维护。
- 长期保存格式策略、格式迁移、派生件治理。
- 开放鉴定、销毁鉴定、销毁清册和审批。
- 阅览室、借阅、复制、利用统计和利用审批。
- 开放目录、公众查询、脱敏发布和版权控制。
- 编研、专题库、展陈、档案资源开发。
- 监管报表、质量分析、容量预测、灾备演练。
- 跨系统接入：OA、ERP、财务、合同、人事、科研、教务、影像、邮件、网页和社交媒体。

### 验收口径

- 能支撑档案从形成、归档、保管、利用、开放、鉴定、移交到销毁或永久保存的全生命周期。
- 能应对机构沿革、分类调整、法规变化、存储迁移和系统升级。
- 能提供可靠的证据链、审计链和恢复能力。

## 建议的下一批 OpenSpec 变化

如果要继续做基础能力，不建议一次创建一个巨大 change。建议拆成可以独立验收的序列：

1. `add-archive-classification-schemes`  
   分类方案、分类节点版本、模板、分类到动态表和档号规则的关系。

2. `add-archive-reference-code-rules`  
   档号规则、组成项、流水、唯一性、手工调整和档号历史。

3. `add-archive-fonds-governance`  
   立档单位、全宗沿革、全宗关系、全宗和权限/分类方案关系。

4. `add-archive-retention-schedules`  
   归档范围、保管期限表、到期提醒和处置动作。

5. `add-archive-filing-workflow`  
   收集、预检、整理、著录、审核、入库、退回和问题清单。

6. `add-archive-file-components`  
   主文件、附件、组件、派生件、版本、格式和摘要关系。

7. `add-archive-description-catalogs`  
   著录项、案卷目录、卷内目录、文件级目录和专题目录。

8. `add-archive-quality-checks`  
   四性检测任务、检测项、检测结果和修复记录。

9. `add-archive-transfer-accession`  
   移交批次、移交清单、接收检测、问题反馈和回执。

10. `add-archive-use-services`  
    查档、阅览、借阅、下载、复制、导出、利用登记和利用审批。

## 近期推荐路线

当前最值得先做的是“基础治理层”，不是完整平台：

```text
分类方案
   │
   ├── 绑定全宗适用范围
   ├── 绑定动态表/字段集合
   └── 绑定档号规则

全宗治理 ───────▶ 数据范围 / 来源追溯

档号规则 ───────▶ 档案可引用、可排列、可移交

保管期限表 ────▶ 归档范围 / 到期处置

归档流程 ───────▶ 正式入库 / 问题清单 / 审计
```

第一阶段可以只提一个 OpenSpec change：`add-archive-governance-foundation`，把分类方案、全宗治理、档号规则、保管期限表的最小合同放进去。实现时再按模块拆任务。

如果担心 scope 太大，就拆成前三个 change：分类方案、档号规则、全宗治理。三者是最容易互相影响的地基。
