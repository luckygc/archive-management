# 归档、移交与单套管理

本页关注电子文件归档、电子档案移交接收和电子档案单套管理，主要参考 `GB/T 18894-2016`、`DA/T 85-2019`、`DA/T 92-2022`、`DA/T 93-2022` 和 `DA/T 98-2023`。

## 术语边界

- **档案**：归档后纳入档案管理体系的对象，是名词。
- **归档**：把具有保存价值的电子文件及其元数据、组件、过程信息纳入档案管理的动作或流程。
- **移交**：形成单位、业务系统或管理机构向档案管理系统或档案馆转交档案及其相关信息。
- **接收**：档案管理系统对移交内容进行校验、登记、入库和反馈。
- **单套管理**：电子档案作为唯一或主档案载体管理时，需要更严格的可信、可控和可验证措施。

项目命名上建议继续区分：

- `Archive` 表示档案领域名词。
- `file` 表示执行归档动作。
- `Filing` 表示归档流程、归档业务或归档申请。
- `archived` / `filed` 表示已归档状态，需要按业务语境固定一个词。

## 归档流程理解

归档不是简单上传文件。一个较完整的电子归档链路通常包括：

1. 确认归档范围和归档责任。
2. 收集电子文件、组件、附件、元数据和过程信息。
3. 校验文件格式、摘要、完整性、必填元数据和业务一致性。
4. 建立档案分类、全宗、案卷、件、档号和保管期限等档案关系。
5. 写入档案库、对象存储、索引和审计记录。
6. 输出归档结果、问题清单或回执。

## 移交接收理解

移交通常比普通导入更强调“包”和“回执”：

- 移交方需要说明移交范围、批次、来源系统、责任人和文件清单。
- 接收方需要检测元数据、文件、包结构、摘要、数量和关联关系。
- 问题应形成可追踪的反馈，不应只返回一条笼统失败消息。
- 接收成功后，应能查询批次、档案、文件对象和检测结果。

对当前项目的影响：

- 批量导入、外部系统接入和档案移交不应只复用同一个“上传 Excel”语义。
- 如果做正式移交，需要独立建模移交批次、接收批次、问题明细和回执。
- 异步任务应按 `api-contract` 的 LRO 口径返回 job，并能查询处理进度和结果。

## 单套管理理解

单套管理不是一个字段开关，而是一组更严格的管理要求。项目上至少要考虑：

- 归档前能证明电子文件来源可信。
- 归档时能校验文件和元数据完整。
- 归档后能防止未授权修改，并记录所有关键操作。
- 能长期读取、展示、导出或迁移文件格式。
- 能备份、恢复，并能证明恢复后内容未被破坏。

## 建议进入 OpenSpec 的能力

| 能力                       | 建议落点                               |
| -------------------------- | -------------------------------------- |
| 归档申请和归档状态机       | 新增或扩展业务规格，不写在 `AGENTS.md` |
| 移交批次、移交包和接收回执 | 新增移交接收规格                       |
| 归档检测结果和问题清单     | 归档/移交规格中的响应和异步任务结果    |
| 单套管理声明               | 单独规格，明确支持范围和验收条件       |
| 档案来源和业务系统映射     | 档案元数据或接入规格                   |

## 待确认

- 项目是否要把“归档”作为核心流程，还是先只做档案库管理。
- 是否要支持正式移交包和接收回执。
- “已归档”状态在当前模型中表示业务归档完成，还是表示实体/电子文件进入档案库。
- 是否声明支持电子档案单套管理；如果支持，需要补充更严格的检测、审计、备份和防篡改能力。
