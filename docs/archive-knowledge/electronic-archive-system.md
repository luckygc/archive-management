# 电子档案系统能力

本页关注电子文件管理系统和电子档案管理系统的通用功能要求，主要参考 `GB/T 29194-2012`、`GB/T 39784-2021`、`GB/T 31914-2015` 和电子档案管理系统相关规定。

## 理解框架

可以把系统能力拆成两段：

- **电子文件管理系统**：偏向文件形成、办理、捕获、分类、处置和向档案阶段移交。
- **电子档案管理系统**：偏向归档后接收、整理、保存、利用、统计、移交、备份和长期维护。

当前项目名称是档案管理系统，更接近第二段，但如果要接业务系统归档，仍需要理解第一段的输入边界。

## 功能域映射

| 标准能力域     | 项目理解                                               | 当前/后续落点                                        |
| -------------- | ------------------------------------------------------ | ---------------------------------------------------- |
| 收集与接收     | 从业务系统、导入包、人工录入或批量接口接收文件和元数据 | 导入、接口接入、对象存储、元数据映射                 |
| 分类与整理     | 按全宗、分类、案卷、件、保管期限等组织档案             | `archive-metadata`、`archive item`、`archive volume` |
| 著录与元数据   | 维护业务字段、档案元数据、系统元数据和过程元数据       | 动态字段、字段布局、搜索投影                         |
| 检索与利用     | 支持目录检索、全文检索、条件检索、借阅或下载           | `archive-record-search`、权限控制、导出              |
| 权限与安全     | 控制用户、角色、数据范围、操作权限和敏感信息           | 认证授权、数据范围、审计                             |
| 日志与审计     | 记录关键操作、状态变更和利用行为                       | 审计表、业务流水、系统日志                           |
| 存储与长期保存 | 管理文件对象、格式、校验、备份和迁移                   | `file-storage`、校验、备份恢复                       |
| 统计与报表     | 输出档案数量、状态、保管期限、利用和移交统计           | 后续报表规格                                         |

## 对当前项目的设计启发

### 不把系统能力压成单表 CRUD

电子档案系统的核心不是“档案记录表 + 附件表”。更稳定的边界是：

- 档案业务身份：分类、全宗、案卷、条目、档号、保管期限。
- 文件对象身份：存储对象、格式、大小、摘要、版本、来源。
- 元数据身份：字段定义、值、布局、搜索投影、导入映射。
- 过程身份：归档、移交、接收、检测、修复、利用、导出和审计。

### 目录检索和全文检索要分层

目录检索通常依赖档案元数据和业务字段；全文检索依赖正文、OCR、附件解析和搜索 provider。项目里应继续保持 `archive-record-search` 对 provider 的抽象，不把业务查询核心绑定到具体全文引擎。

### 权限需要覆盖“看得见”和“做得了”

系统功能要求通常同时关注查询、读取、下载、修改、删除、移交、导出、统计等操作。项目设计应区分：

- 菜单或按钮是否可见。
- 接口是否允许调用。
- 当前用户是否有目标档案的数据范围。
- 当前档案状态是否允许该操作。

## 建议进入 OpenSpec 的能力

- 电子档案系统的基础能力清单和分阶段验收范围。
- 档案接收、整理、著录、检索、利用、导出、移交流程。
- 文件对象、元数据、检测结果、审计记录之间的关系。
- 权限、数据范围和状态机对操作的约束。

## 暂不直接采用

- 不因为标准列出完整能力，就一次性实现所有统计、鉴定、销毁、开放利用和移交功能。
- 不引入通用“大档案系统适配层”来预留所有未来场景。
- 不把电子文件形成阶段的要求强行塞进当前档案库，除非明确要接业务系统前置办理链路。
