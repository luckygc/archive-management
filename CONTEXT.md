# 领域上下文

本文统一 archive-management 中反复使用的领域词汇和概念关系，帮助开发者与工程技能使用一致语言。它不定义字段、权限、状态机或验收要求；发生冲突时，以对应 `openspec/specs/` 和活动 change 为准。

## 产品边界

archive-management 是面向机构内部使用的档案管理系统，覆盖档案配置、档案记录、归档接收、审批待办、文件管理和系统管理。PC 工作台服务档案管理人员、业务部门经办人和系统管理员，不承担营销展示或公共内容发布。

## 核心词汇

| 词汇 | 含义 | 避免混用 |
| --- | --- | --- |
| 全宗 | 由一个独立机关、组织或个人形成的档案整体，也是分类适用范围和档案数据范围的重要维度 | 不用“租户”代替；当前系统没有以全宗等同租户的合同 |
| 分类方案 | 组织分类节点的一套方案，可限定全宗可使用的分类范围 | 不与单个“分类节点”混用 |
| 分类节点 | 分类方案中的具体分类位置，决定档案对象的动态字段、动态表和部分治理配置 | 不简称为“目录”，避免与文件目录混淆 |
| 档案条目 | 独立的档案记录，保存固定字段并按分类扩展动态字段，可以归入案卷、关联电子文件和其他条目 | 不泛称“文档”或“文件” |
| 案卷 | 独立于档案条目的聚合对象，用于组织一组档案条目 | 不把案卷建模为条目的父记录或同表层级 |
| 电子文件 | 与档案对象关联的文件元数据及其对象存储内容 | “文件记录”指元数据，“对象”指存储内容，二者不混为一物 |
| 归档接收 | 外部或业务数据进入正式档案核心前的接收过程 | 尚未完成归档的数据不称为正式档案条目 |
| 治理方案 | 一组适用范围、运行时约束和规则的逻辑集合 | 不与具体可执行的“治理版本”混用 |
| 治理版本 | 治理方案的版本化配置；草稿可编辑，发布后按状态机保持不可变 | 不用部署版本、应用版本代替 |
| 运行时约束 | 在固定触发点校验候选业务数据，可产生阻断或警告决策 | 不称为任意脚本；系统不执行用户 SQL 或脚本 |
| 运行时规则 | 在固定触发点根据结构化条件执行系统支持的固定动作 | 不与数据库约束或前端表单规则混用 |
| 审批流程定义 | 描述审批节点、连线和受控条件分支的可发布定义 | 不与某次运行中的“审批流程实例”混用 |
| 审批流程实例 | 基于已发布定义启动的一次审批过程 | 运行态任务、候选关系和历史以 Flowable 为准 |
| 统一待办 | 聚合不同来源业务任务的可重建查询投影 | 不作为来源业务状态或最终授权的真相源 |
| 功能权限 | 决定用户能否执行某类功能或访问某类入口 | 不等同于档案数据范围 |
| 档案数据范围 | 决定用户可以读取或修改哪些档案对象 | 不能用前端可见性或功能权限替代 |
| 动态字段 | 由分类配置定义、用于扩展档案对象的业务字段 | 不把固定系统字段重复定义为动态字段 |
| 动态表 | 按稳定业务键为分类和对象类型生成的 PostgreSQL 表 | 不允许未经白名单校验的动态标识符 |

## 概念关系

- 全宗选择可用的分类方案或分类范围；分类节点属于且只能属于一个分类方案。
- 档案条目和案卷是不同对象，分别持久化；条目可以加入案卷，但不通过主表层级字段表达两者关系。
- 分类节点决定档案条目、案卷或明细行可使用的动态字段和动态表。
- 档案对象可以关联电子文件；文件内容存放在 S3 兼容对象存储，数据库保存文件元数据和业务关联。
- 新建档案保存当时解析出的治理版本，使历史档案继续按其绑定版本解释。
- 运行时约束和规则由治理版本装配，在业务副作用发生前执行；事务仍由对应业务用例拥有。
- 审批流程产生的任务可以投影到统一待办，但办理时必须回到审批来源校验状态和权限。
- 功能权限、档案数据范围和来源业务授权共同构成访问边界，三者不可相互替代。

## 关键业务不变量

- 已发布的治理版本和审批流程定义不可原地修改；变更通过新版本表达。
- 归档写入、运行时规则动作和相关审计必须处于可追踪的事务边界，失败时不得留下部分写入。
- 服务端权限、数据范围、状态和数据库约束是最终边界；前端控制只改善体验。
- 动态表名和动态列必须来自经过校验的配置，项目 SQL 只面向 PostgreSQL。
- 统一待办、搜索投影和缓存均为可重建派生数据，不能替代来源业务真相。
